Dynamo is a big system — disaggregated serving, KV-aware routing, multi-tier caching, autoscaling, Kubernetes operators. The reference docs cover every knob, but a newcomer doesn't want every knob. They want **one path that is known to work.**

A **well-lit path** is exactly that: a tested, opinionated, end-to-end route through Dynamo. Each stage builds on the one before it, every command is meant to be copy-pasted and run, and each stage ends with a working system you can actually send traffic to. When you need more, the deep-dive docs are one click away — but you never *have* to leave the path to get something running.

<Note>
**You don't have to walk the whole path.** Each stage is a complete, working deployment. Stop at the stage that meets your needs — a lot of teams run happily on Stage 1 or 2 and never disaggregate. The later stages are there when your scale demands them, not as homework.
</Note>

## The Path

  <Card title="Stage 0 · Hello World — one request in 5 minutes" icon="regular bolt" href="/dynamo/dev/getting-started/quickstart">

  </Card>
  <Card title="Stage 1 · Run a Real Model as a Service" icon="regular server" href="/dynamo/dev/well-lit-paths/aggregated-service">

  </Card>
  <Card title="Stage 2 · Add KV-Aware Routing" icon="regular route" href="/dynamo/dev/well-lit-paths/kv-routing">

  </Card>
  <Card title="Stage 3 · Disaggregate Prefill and Decode" icon="regular arrows-split-up-and-left" href="/dynamo/dev/well-lit-paths/disaggregated">

  </Card>
  <Card title="Stage 4 · Go to Production on Kubernetes" icon="regular cubes" href="/dynamo/dev/well-lit-paths/production">
    Take the same topology to a cluster: install the Dynamo Platform, deploy with one manifest, let AIConfigurator pick the config and the Planner autoscale to your SLA. Add observability and fault tolerance. **Hours · multi-GPU / multi-node.**
  </Card>

## From 1 to 100

The stages are deliberately ordered so each one adds a single, learnable idea. You go from *"can I get a response"* to *"can I run this for real users at scale"* without ever facing the whole system at once.

| Stage | You go from… | …to | Adds | Hardware |
|:---|:---|:---|:---|:---|
| **0 · Hello World** | nothing | one request answered | a frontend + a worker | 1 GPU |
| **1 · Real Service** | a toy model | a real model on a stable endpoint | persistence, health checks, model choice | 1–2 GPUs |
| **2 · KV Routing** | one worker | many workers, routed by cache overlap | the router (prefix reuse) | 2+ GPUs |
| **3 · Disaggregation** | one worker type | independent prefill + decode pools | NIXL KV transfer | 2+ GPUs |
| **4 · Production** | a local process | a Kubernetes service that scales to SLA | Platform, DGDR, Planner, observability | multi-node |

## Is Dynamo even the right tool?

Dynamo is an orchestration layer **above** SGLang, TensorRT-LLM, and vLLM — it coordinates them, it doesn't replace them. It earns its keep when you're serving across **multiple GPUs or nodes**, want **KV-aware routing**, need to **scale prefill and decode independently**, or need **SLA-driven autoscaling**.

> [!TIP]
> If you're running a single model on a single GPU with no scaling needs, your inference engine alone is probably enough — and Stage 0 will show you that in five minutes. Come back when you outgrow it.

## Conventions used on every stage

- **What you'll have at the end** — stated up front, so you know where you're going.
- **Prerequisites** — hardware, time, and which earlier stage to finish first.
- **Copy-paste commands** — every block is runnable as written; placeholders are in `UPPER_CASE`.
- **Verify it works** — a concrete check before you move on.
- **Stop here if… / Continue when…** — explicit guidance on whether the next stage is worth it for you.
- **Go deeper** — links into the reference docs for the knobs this stage intentionally left at their defaults.
