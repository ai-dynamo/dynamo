## What you'll have at the end

The same routed, disaggregated service from Stage 3 — but running on Kubernetes as a declarative deployment that **autoscales to an SLA**, **survives worker failures**, and **exports metrics, logs, and traces**. You describe the model, hardware, and latency targets in one manifest; Dynamo profiles, configures, and deploys it.

## Prerequisites

- A **Kubernetes cluster (v1.24+)** with GPU nodes, plus [kubectl](https://kubernetes.io/docs/tasks/tools/#kubectl) and [Helm](https://helm.sh/docs/intro/install/) (v3+).
- The [NVIDIA GPU Operator](https://docs.nvidia.com/datacenter/cloud-native/gpu-operator/latest/getting-started.html) installed (the only hard prerequisite — see the [Installation Guide](/dynamo/dev/kubernetes-deployment/deployment-guide/installation-guide) for a one-liner).
- A HuggingFace token if any model is gated.

## What changes when you go to Kubernetes

Locally you ran processes by hand. On a cluster, three things take that over:

- The **Dynamo Operator** turns a single custom resource into the right set of frontend, prefill, and decode pods.
- The **DynamoGraphDeploymentRequest (DGDR)** is your entrypoint: you state *what* (model + hardware + SLA), it profiles your workload, picks a config, and creates the running deployment.
- The **Planner** watches live traffic and scales prefill/decode pools to hold your SLA — the autoscaling you couldn't get from hand-launched processes.

## Step 1 — Create the HuggingFace secret

```bash
export HF_TOKEN=<your-hf-token>
kubectl create secret generic hf-token-secret --from-literal=HF_TOKEN="$HF_TOKEN"
```

## Step 2 — Install the Dynamo Platform

```bash
export NAMESPACE=dynamo-system
helm install dynamo-platform \
  oci://helm.ngc.nvidia.com/nvidia/ai-dynamo/charts/dynamo-platform \
  --version "1.0.2" \
  --namespace "$NAMESPACE" --create-namespace
```

```bash
kubectl get pods -n $NAMESPACE
# Expect dynamo-operator-*, etcd-*, and nats-* pods Running
```

> [!NOTE]
> Check [Release Artifacts](/dynamo/dev/resources/release-artifacts) for the current chart and image versions before you deploy.

## Step 3 — Deploy with one manifest (zero-config)

Describe the model, backend, and SLA. AIConfigurator profiles it, the Planner optimizes the topology, and Dynamo deploys — no hand-written worker configs.

```yaml
# my-model.yaml
apiVersion: nvidia.com/v1beta1
kind: DynamoGraphDeploymentRequest
metadata:
  name: my-model
spec:
  model: Qwen/Qwen3-0.6B
  backend: auto          # let the profiler choose vllm / sglang / trtllm
  sla:
    ttft: 200.0          # ms — time to first token
    itl: 20.0            # ms — inter-token latency
  autoApply: true
```

```bash
kubectl apply -f my-model.yaml -n $NAMESPACE

# Watch it move Pending → Profiling → Deploying → Deployed
kubectl get dgdr my-model -n $NAMESPACE -w
```

## Step 4 — Send a request

```bash
FRONTEND_SVC=$(kubectl get svc -n $NAMESPACE -o name | grep frontend | head -1)
kubectl port-forward "$FRONTEND_SVC" 8000:8000 -n $NAMESPACE &

curl -s localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "Qwen/Qwen3-0.6B",
       "messages": [{"role": "user", "content": "Hello from Kubernetes!"}],
       "max_tokens": 50}' | jq
```

You now have the full topology — frontend, routing, and (for larger models) disaggregated prefill/decode — running as managed Kubernetes workloads.
