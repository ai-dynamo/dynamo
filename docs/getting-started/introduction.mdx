Dynamo is an open-source, high-throughput, low-latency inference framework for serving generative AI workloads in distributed environments. It is Kubernetes-native for production, while the same frontend, router, and worker stack also runs in local containers for evaluation or incremental adoption.

This User Guide covers installing, deploying, and operating Dynamo on Kubernetes. New here? Pick a quickstart below.

## Get Started

Dynamo is **Kubernetes-native, but not Kubernetes-only**. The full feature set — operator, CRDs, autoscaling, inference gateway, and multinode scheduling — runs on Kubernetes, while the same frontend, router, and worker stack also runs locally for evaluation. This guide assumes a Kubernetes cluster; to run in a local environment or build from source, see [Local Deployment](#local-deployment).

## Deploy

Deploying a model means describing an inference graph — a Frontend plus one or more workers — as a Kubernetes custom resource, then letting the Dynamo operator reconcile it into running pods. There are two paths: author a **DynamoGraphDeployment (DGD)** by hand for full control, or describe intent with a **DynamoGraphDeploymentRequest (DGDR)** and let the profiler size it for you. Start with the [Model Deployment overview](/dynamo/dev/getting-started/kubernetes-deployment) to choose between them.

### Deploy with DGD

Author the spec — components, parallelism, replicas — and `kubectl apply` it. A DGD works for any model and backend, and is the object that serves traffic.
