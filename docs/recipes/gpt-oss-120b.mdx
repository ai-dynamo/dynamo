## Deployment Instructions

<div id="deploy"></div>

First prepare the namespace, HuggingFace token secret, and model cache (shared by both targets):

```bash
export NAMESPACE=dynamo-demo
kubectl create namespace ${NAMESPACE}

kubectl create secret generic hf-token-secret \
  --from-literal=HF_TOKEN="your-token-here" \
  -n ${NAMESPACE}

# Update storageClassName in model-cache/model-cache.yaml first!
kubectl apply -f recipes/gpt-oss-120b/model-cache/ -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=3600s
```

<h3 id="deploy-tensorrt-llm-aggregated">TensorRT-LLM aggregated</h3>

WideEP aggregated serving for short-prompt, high-concurrency throughput.

```bash
kubectl apply -f recipes/gpt-oss-120b/trtllm/agg/deploy.yaml -n ${NAMESPACE}
```

<h3 id="deploy-tensorrt-llm-disaggregated">TensorRT-LLM disaggregated</h3>

Prefill/decode split for long-context generation. Model loading takes roughly 15-30 minutes depending on storage speed:

```bash
kubectl apply -f recipes/gpt-oss-120b/trtllm/disagg/deploy.yaml -n ${NAMESPACE}
kubectl get pods -n ${NAMESPACE} -l app.kubernetes.io/part-of=gpt-oss-disagg -w
```

## Smoke Test

<div id="smoke-test"></div>

Run a small request before starting the benchmark. The frontend service is `gpt-oss-agg-frontend` for the aggregated target and `gpt-oss-disagg-frontend` for the disaggregated target.

```bash
export NAMESPACE=dynamo-demo
kubectl port-forward svc/gpt-oss-agg-frontend 8000:8000 -n ${NAMESPACE}

curl http://localhost:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"openai/gpt-oss-120b","messages":[{"role":"user","content":"Write a one-sentence readiness check."}],"max_tokens":64}'
```

## Benchmark Instructions

<div id="benchmark"></div>

Each target ships a `perf.yaml` Kubernetes Job that waits for the model to come up, then runs AIPerf with that target's traffic shape and a request count of 10x total concurrency. The core AIPerf invocation it wraps:

```bash
aiperf profile \
  --model openai/gpt-oss-120b \
  --endpoint-type chat --endpoint /v1/chat/completions --streaming \
  --url http://<frontend>:8000 \
  --synthetic-input-tokens-mean <isl> --output-tokens-mean <osl> \
  --extra-inputs ignore_eos:true \
  --concurrency <total concurrency> --request-count <10 x concurrency>
```

<h3>TensorRT-LLM aggregated (ISL 128 / OSL 1000, 900 per GPU x 4 GPUs = 3,600 concurrency)</h3>

```bash
kubectl apply -f recipes/gpt-oss-120b/trtllm/agg/perf.yaml -n ${NAMESPACE}
kubectl logs -f -l job-name=gpt-oss-120b-bench -n ${NAMESPACE}
```

<h3>TensorRT-LLM disaggregated (ISL 8192 / OSL 1024, 1,536 concurrency)</h3>

```bash
kubectl apply -f recipes/gpt-oss-120b/trtllm/disagg/perf.yaml -n ${NAMESPACE}
kubectl logs -f -l job-name=gpt-oss-120b-disagg-bench -n ${NAMESPACE}
```

## Notes

- This recipe requires ARM64 (GB200) nodes for the aggregated target — it will not run on x86 Hopper/Ampere hardware. The disaggregated target accepts GB200 or B200.
- Do not read the two targets as an aggregated-vs-disaggregated benchmark; their traffic shapes differ by design.
- The disaggregated deployment uses 5 GPUs (1x TP1 prefill + 1x TP4 decode), while its `perf.yaml` computes total concurrency from a 6-GPU count (256 x 6 = 1,536); adjust `DEPLOYMENT_GPU_COUNT` if you want strict per-GPU normalization.
- Disaggregated engine configs differ per role: prefill runs TP1 with `max_batch_size=64` and the overlap scheduler disabled; decode runs TP4 with `max_batch_size=1280` and the overlap scheduler enabled. KV transfer uses the UCX-based cache transceiver (`max_tokens_in_buffer=9216`).
- The disaggregated target uses `W4A8_MXFP4_MXFP8` quantization via the `OVERRIDE_QUANT_ALGO` environment variable.
