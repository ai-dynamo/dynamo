This recipe deploys Qwen3.6-35B-A3B-FP8 on a single H100 or GB200 with Dynamo frontend decoding plus an 8 GiB multimodal embedding cache (`dynamo-fd-ec`) — roughly 30% higher RPS than vanilla `vllm serve` on both GPUs. The vanilla vLLM serve and frontend-decoding-only arms are comparison configurations that live on the related Feature Benchmark page.

## Prerequisites

- A Kubernetes cluster with the Dynamo platform installed and **1x H100 or 1x GB200** available.
- A `shared-model-cache` PVC (RWX, e.g. FSx Lustre) in the target namespace — create it out-of-band if your cluster does not pre-provision it.
- `envsubst` on the machine driving the recipe (`apt install gettext-base` on Ubuntu, `brew install gettext` on macOS) — the deploy and perf YAMLs are templates resolved at apply time.
- No Hugging Face token — `Qwen/Qwen3.6-35B-A3B-FP8` is public (`gated: false`). To swap in a gated model, uncomment the `hf-token-secret` blocks in `model-cache/model-download.yaml` and `deploy/<config>.yaml`, then create the secret.

Create the namespace (and the token secret only if you swap in a gated model):

```bash
export NAMESPACE=your-namespace
kubectl create namespace ${NAMESPACE}
# Only needed for gated models — not for Qwen/Qwen3.6-35B-A3B-FP8:
kubectl create secret generic hf-token-secret \
  --from-literal=HF_TOKEN="your-token" \
  -n ${NAMESPACE}
```

<Warning>
Fill in your hostname before first use: edit `hw/h100.env` or `hw/gb200.env` and replace the `<FILL-IN-...-HOSTNAME>` placeholder with a real `kubernetes.io/hostname` from your cluster — all configs pin to one node so runs are comparable. Do not `kubectl apply` the raw YAML files directly; they are `envsubst` templates driven by `run-benchmark.sh`. GB200 nodes additionally need the `kubernetes.io/arch=arm64:NoSchedule` toleration, which `hw/gb200.env` already carries.
</Warning>

Find a node to pin to:

```bash
# H100
kubectl get nodes -L nvidia.com/gpu.product | awk '/H100/'
# GB200
kubectl get nodes -L kubernetes.io/arch -L nvidia.com/gpu.product | awk '/arm64/ && /GB200/'
```

## Deploy

`run-benchmark.sh` sources `hw/${HW}.env`, resolves the templates via `envsubst`, and runs prep (PVC check, model download, dataset generation), deploy, bench, retrieve, and clean:

```bash
cd recipes/qwen3.6-35b
export NAMESPACE=your-namespace
export HW=gb200   # or h100

# Full pipeline for the recommended target
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec

# Or step-by-step (pvc/download/dataset are config-agnostic prep)
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec --step download
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec --step dataset
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec --step deploy
```

The deploy step applies `deploy/dynamo-fd-ec.yaml` — a DynamoGraphDeployment named `qwen36-dynamo-fd-ec` running `dynamo.vllm` with `--frontend-decoding`, `--multimodal-embedding-cache-capacity-gb 8`, and `--mm-processor-cache-gb 30` on TP=1.

## Smoke Test

The Dynamo operator creates the `qwen36-dynamo-fd-ec-frontend` service for this deployment.

```bash
kubectl port-forward svc/qwen36-dynamo-fd-ec-frontend 8000:8000 -n ${NAMESPACE}

curl http://localhost:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"Qwen/Qwen3.6-35B-A3B-FP8","messages":[{"role":"user","content":"Write a one-sentence readiness check."}],"max_tokens":64}'
```

## Benchmark

The shared [`perf.yaml`](https://github.com/ai-dynamo/dynamo/blob/main/recipes/qwen3.6-35b/perf.yaml) at the recipe root wraps this AIPerf run. It is one template for all configs — `run-benchmark.sh` exports `${BENCH_POD}`, `${BENCH_FRONTEND}`, and `${BENCH_RUN_LABEL}` per config and resolves it with `envsubst`, so launch it through the script rather than applying the file directly:

```bash
aiperf profile --model Qwen/Qwen3.6-35B-A3B-FP8 --input-file 30u_8t_5w_8000word_base64.jsonl \
  --custom-dataset-type single_turn --url http://qwen36-dynamo-fd-ec-frontend:8000 --streaming \
  --request-count 240 --concurrency 30 --request-rate-mode constant --warmup-request-count 2 \
  --extra-inputs max_tokens:1024 --extra-inputs min_tokens:1024 --extra-inputs ignore_eos:true
```

```bash
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec --step bench
./run-benchmark.sh -n ${NAMESPACE} --hw ${HW} --config dynamo-fd-ec --step retrieve
```

The retrieve step copies `profile_export_aiperf.json` to `~/workspace/dynamo-tmp/logs/<MM-DD>/qwen36-fp8-${HW}/dynamo-fd-ec/`; throughput, TTFT, and ITL read directly from that file. `run-all-benchmarks.sh` runs all three comparison configs sequentially if you want to reproduce the full Feature Benchmark.

## Expected Performance

Measured with the shared sliding-window workload (concurrency 30, `max_tokens=1024`); deltas are versus the vanilla `vllm serve` baseline on the same pinned node:

**H100**

| Metric | dynamo-fd-ec | Delta vs vllm-serve |
| --- | ---: | ---: |
| RPS | 0.933 | +29.8% |
| TTFT avg | 6,100.92 ms | -66.4% |
| TTFT p50 | 2,369.01 ms | -87.4% |
| ITL | 24.56 ms | +15.4% |

**GB200**

| Metric | dynamo-fd-ec | Delta vs vllm-serve |
| --- | ---: | ---: |
| RPS | 1.236 | +31.5% |
| TTFT avg | 8,478.29 ms | -43.3% |
| TTFT p50 | 8,323.53 ms | -45.9% |
| ITL | 15.22 ms | -1.0% |

ITL stays roughly flat because the embedding cache shortens the prefill path (skipping the vision encoder for repeated images), not decode. The full three-way table including the frontend-decoding-only arm is on the related Feature Benchmark page.

## Notes

- The source intentionally drives everything through `run-benchmark.sh` because one shared `perf.yaml` is parameterized by benchmark pod, frontend service, run label, and hardware env file.
- AIPerf installs from source pinned to a `main` SHA that includes [PR 824](https://github.com/ai-dynamo/aiperf/pull/824); the dataset writes `session_id=user_<N>` per row, and PR 824 makes `single_turn` mode honor that ordering so prefix-cache hits across a user's 8 turns are real. The pin lives in `perf.yaml` (`AIPERF_GIT_REF`).
- Frontend decoding does CPU-side image decode and base64 parsing, so the Frontend gets more CPU (4-8 cores) than a typical frontend pod.
- The related Feature Benchmark includes the vanilla `vllm serve` (`vllm-serve`) and frontend-decoding-only (`dynamo-fd`) arms used to isolate each technique; they are not recipe targets.
- All resources carry a `qwen36-` prefix and `app.kubernetes.io/name: qwen3.6-35b` labels for easy discovery in shared namespaces.
