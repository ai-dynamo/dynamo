## Prerequisites

<div className="dynamo-prereq-grid">
<div><span>Hardware</span><strong>8x B200 or 8x H200 (two 4-GPU worker replicas)</strong></div>
<div><span>Runtime</span><strong>nvcr.io/nvidia/ai-dynamo/vllm-runtime:1.3.0-nemotron-super-dev.1 (vLLM 0.21.0)</strong></div>
<div><span>Dataset</span><strong>Bundled Mooncake-format JSONL traces (15%, 30%, full)</strong></div>
<div><span>Traffic</span><strong>Chat 8K/1K and agentic 64K/400 trace replays</strong></div>
<div><span>Scheduler</span><strong>Namespace labeled kai.scheduler/enabled=true</strong></div>
<div><span>Setup assets</span><strong>model-cache/model-cache.yaml, model-download.yaml, model-download-fp8.yaml</strong></div>
</div>

You need the Dynamo Platform installed on the target cluster (DGD CRDs registered with `nvidia.com/v1beta1` served) and a Hugging Face token secret named `hf-token-secret` with access to the NVFP4 (B200) and/or FP8 (H200) checkpoints.

<Warning>
Label the namespace for KAI before deploying — without `kai.scheduler/enabled=true`, pods sit `SchedulingGated` indefinitely because KAI's `pod-grouper` filters by namespace label. Also edit storage class, image tags, and other cluster-specific settings before applying these manifests.
</Warning>

## Deployment Instructions

<div id="deploy"></div>

Create the namespace with the KAI label, the model-cache PVC, and download the checkpoint for your SKU:

```bash
export NAMESPACE=your-namespace
kubectl create namespace ${NAMESPACE}
kubectl label namespace ${NAMESPACE} kai.scheduler/enabled=true

kubectl create secret generic hf-token-secret \
  --from-literal=HF_TOKEN="$HF_TOKEN" \
  -n ${NAMESPACE}

# Edit storageClassName in model-cache.yaml first (kubectl get storageclass)
kubectl apply -f recipes/nemotron-3-super/model-cache/model-cache.yaml -n ${NAMESPACE}

# B200 — NVFP4 checkpoint (~80 GB)
kubectl apply -f recipes/nemotron-3-super/model-cache/model-download.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=1800s

# H200 — FP8 checkpoint (~120 GB)
kubectl apply -f recipes/nemotron-3-super/model-cache/model-download-fp8.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download-fp8 -n ${NAMESPACE} --timeout=3600s
```

Downloading both checkpoints lands ~200 GB on the PVC — the default size in `model-cache.yaml` — so bump `storage:` before downloading both.

### B200 and H200 × chat/agentic

Pick the SKU and use-case variant and apply. The DGD is named `nemotron-3-super-<sku>-<usecase>`:

```bash
SKU=b200       # or h200
USECASE=chat   # or agentic

kubectl apply -f recipes/nemotron-3-super/vllm/agg-${SKU}-${USECASE}/deploy.yaml -n ${NAMESPACE}
kubectl get dgd nemotron-3-super-${SKU}-${USECASE} -n ${NAMESPACE} -w
```

First-time boot per worker takes about 6–9 minutes (image pull + vLLM engine init + Inductor + CUDA graph capture up to size 512).

Each DGD exposes a frontend service named `nemotron-3-super-<sku>-<usecase>-frontend`. B200 serves the NVFP4 model id; H200 serves the FP8 model id:

```bash
kubectl port-forward svc/nemotron-3-super-${SKU}-${USECASE}-frontend 8000:8000 -n ${NAMESPACE}

  -H "Content-Type: application/json" \

       \"max_tokens\":64,

```

## Benchmark Instructions

<div id="benchmark"></div>

A single AIPerf trace-replay Job, `perf/perf.yaml`, covers all four targets. Edit `ENDPOINT`, `TRACE_FILE`, `TARGET_MODEL`, and `CONCURRENCY` in the Job env block per variant (`TARGET_MODEL` is NVFP4 for B200, FP8 for H200). The Job runs this AIPerf Mooncake trace replay:

```bash
aiperf profile \
  -m nvidia/NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4 \
  --tokenizer nvidia/NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4 \
  --tokenizer-trust-remote-code \
  --input-file /model-cache/traces/8k_1k_70kv_chat_new_noschedule.jsonl \
  --custom-dataset-type mooncake_trace \
  --prompt-input-tokens-block-size 512 \
  --url http://nemotron-3-super-b200-chat-frontend:8000 \
  --streaming --use-server-token-count \
  --extra-inputs ignore_eos:true \
  --concurrency 24 --random-seed 42 \
  --export-http-trace
```

Stage the bundled traces on the `model-cache` PVC under `/model-cache/traces/` (see the [perf README](https://github.com/ai-dynamo/dynamo/blob/main/recipes/nemotron-3-super/perf/README.md) for the helper-pod workflow), then run the Job:

```bash
kubectl apply -f recipes/nemotron-3-super/perf/perf.yaml -n ${NAMESPACE}
kubectl logs -n ${NAMESPACE} -l job-name=nemotron-3-super-bench -f
kubectl wait --for=condition=Complete job/nemotron-3-super-bench -n ${NAMESPACE} --timeout=7200s
```

Artifacts land on the PVC under `/model-cache/perf/<epoch>_nemotron-3-super-bench/`. For concurrency sweeps, delete the worker pods between runs so residual KV/prefix-cache state does not skew results (Dynamo workers are PodClique pods — `kubectl rollout restart deployment` is a silent no-op).

## Notes

- The namespace must carry the `kai.scheduler/enabled=true` label before deploying; without it, pods stay `SchedulingGated` indefinitely.
- B200 chat and agentic ship with MTP spec dec ON by default (DL=3, `moe_backend=triton`, stripped `compilation-config`, `MAX_NUM_BATCHED_TOKENS=65536`). To turn MTP off, remove the `- --speculative-config=$(SPECULATIVE_CONFIG)` line from worker args; with the freed memory headroom you can bump `MAX_NUM_BATCHED_TOKENS` to `"131072"` and switch `COMPILATION_CONFIG` to the `compilation-config-fused` ConfigMap key for better throughput.
- For a fixed-AL synthetic MTP run on H200, point the `SPECULATIVE_CONFIG` env at the `speculative-config-synthetic` ConfigMap key before deploying.
- Known issue: some 400 HTTP errors raised by the workers on invalid inputs surface as 500 through the Dynamo frontend (the proxy does not always preserve the worker's original status code).
- Reasoning is controlled per request via `chat_template_kwargs` (`enable_thinking: true|false`); tool calling and function calling with JSON arguments are supported.
- The recipe runs on a dedicated runtime image (`vllm-runtime:1.3.0-nemotron-super-dev.1`, vLLM 0.21.0).
