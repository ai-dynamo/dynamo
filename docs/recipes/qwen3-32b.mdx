## When to Use

Use this recipe when requests have long prompts, repeated prefixes or session history, and shorter
generated outputs. The KV router targets TTFT by routing compatible requests to workers with useful
cached blocks; the prefill/decode split targets ITL by reducing decode interruption from long
prefill work. Keep the aggregated round-robin variant as the control run for your own cluster.

## Configuration Notes

<Accordion title="Disagg + KV router">
The frontend runs with KV routing enabled. Prefill and decode workers both use the NIXL KV connector,
TP2, 64-token blocks, async scheduling, and a 131K max model length with YaRN rope scaling. Prefill
workers publish KV cache events so the router can account for useful cached blocks. The manifest
requests RDMA resources for both prefill and decode workers.
</Accordion>

<Accordion title="Aggregated round-robin">
The aggregate baseline keeps the same model, TP2, block size, async scheduling, max model length, and
YaRN rope scaling, but removes the prefill/decode split and KV-aware routing. Treat this as the
control run when you compare Dynamo behavior on your own cluster.
</Accordion>

<Accordion title="Aggregated + KVBM">
The KVBM variant uses the Dynamo KVBM connector through vLLM's `--kv-transfer-config` and sets
`DYN_KVBM_CPU_CACHE_GB=100`. The worker asks for 1 GPU plus 150 GiB of host memory, with a 200 GiB
limit. Metrics are off by default; enable `DYN_KVBM_METRICS=true` and expose port `6880` if you want
Prometheus counters such as `kvbm_offload_blocks_d2h`, `kvbm_onboard_blocks_h2d`,
`kvbm_matched_tokens`, and `kvbm_host_cache_hit_rate`.
</Accordion>

## Why This Recipe

This workload has long input sequences, shorter outputs, and meaningful prefix reuse. KV-aware
routing can send requests to workers that already hold matching KV blocks, reducing redundant
prefill. Disaggregated serving separates prefill and decode workers, which helps avoid decode
interruption from long prefill requests.
