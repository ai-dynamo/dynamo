## When to Use

Use this recipe when GLM-5 traffic is dominated by long generated outputs and you can dedicate a
large decode rank group. The benchmark stresses 8K output tokens at 512 concurrent users, so TTFT
includes queueing pressure; use it to evaluate decode throughput and interactivity under burst load,
not as a low-concurrency latency estimate.

## Results

Reference run: 1,536 requests, 0 errors, and 747.87 seconds of benchmark duration.

| Metric | Value |
| --- | ---: |
| Output throughput | 16,824 tokens/sec |
| Request throughput | 2.05 requests/sec |
| TTFT p50 | 15,423 ms |
| ITL avg | 23.31 ms/token |
| Tokens/user/sec avg | 43.39 |
| Request errors | 0 |

<Note>
The benchmark is a concurrency burst at 512 users, so TTFT includes queueing. Re-run the provided
`perf.yaml` on your own cluster before using these numbers for sizing.
</Note>

## Configuration Notes

<Accordion title="EAGLE speculative decoding">
The deployment enables SGLang's EAGLEWorkerV2 path with `SGLANG_ENABLE_SPEC_V2=1`.
`SGLANG_NVFP4_CKPT_FP8_NEXTN_MOE=1` quantizes the BF16 MTP layer to FP8 at load time so it matches
the base model compute path. The recipe README reports roughly 85-95% speculative accept rate.
</Accordion>

<Accordion title="KV cache and long outputs">
The worker uses `--kv-cache-dtype fp8_e4m3`, which cuts KV memory roughly in half compared with
BF16 KV. This matters for the 8K-output benchmark and the large decode group.
</Accordion>

<Accordion title="NIXL / UCX transfer">
Disaggregated KV transfer uses NIXL with UCX. The manifest sets
`UCX_TLS=cuda_copy,cuda_ipc,tcp`: CUDA IPC/copy handles GPU memory movement, while TCP carries UCX
active-message control traffic.
</Accordion>

<Accordion title="FlashInfer JIT cache">
The runtime includes FlashInfer, but the recipe writes first-run JIT artifacts to the shared model
PVC with `FLASHINFER_WORKSPACE_BASE=/model-store` so later pod starts can reuse them.
</Accordion>

<Accordion title="Recovery and rollouts">
The decode side is one TP16 rank group spread across four GB200 nodes. Treat single decode-pod
replacement as disruptive until your cluster has validated full-group recovery; the recipe notes
that frontend and prefill replacement recovered in validation, while individual decode deletion
kept the graph NotReady during repeated rank-group reinitialization attempts.
</Accordion>
