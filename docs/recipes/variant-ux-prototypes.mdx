## Fit Matrix

<table className="dynamo-prototype-fit-table">
  <thead>
    <tr>
      <th>Recipe family shape</th>
      <th>Option 1: table chooser</th>
      <th>Option 2: guided picker</th>
      <th>Option 3: variant subpages</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Single canonical recipe</td>
      <td>Good: table collapses away.</td>
      <td>Poor: no real control exists.</td>
      <td>Acceptable but heavy.</td>
    </tr>
    <tr>
      <td>Recommended vs control</td>
      <td>Excellent: compact comparison.</td>
      <td>Acceptable if the control is a real selection.</td>
      <td>Good when commands diverge heavily.</td>
    </tr>
    <tr>
      <td>Benchmark comparison set</td>
      <td>Excellent: one table communicates evidence.</td>
      <td>Risky: users may expect unsupported combinations.</td>
      <td>Good but adds navigation weight.</td>
    </tr>
    <tr>
      <td>Hardware/runtime matrix</td>
      <td>Good, but wide tables can get dense.</td>
      <td>Excellent if generated from metadata.</td>
      <td>Excellent for deep deploy pages.</td>
    </tr>
    <tr>
      <td>Feature toggle comparison</td>
      <td>Excellent for two-way A/B results.</td>
      <td>Acceptable only for true toggles.</td>
      <td>Usually overkill.</td>
    </tr>
    <tr>
      <td>Deploy-only examples</td>
      <td>Good with clear status badges.</td>
      <td>Awkward unless separated from benchmarks.</td>
      <td>Excellent when build/deploy steps are long.</td>
    </tr>
  </tbody>
</table>

<Note>
Prototype read: Option 1 is the strongest default for the revised recipes page. Option 3 is the
best long-term documentation architecture if the recipe set keeps expanding. Option 2 is attractive
for matrix families, but it needs generated metadata and strict unavailable-state handling to avoid
becoming another fake selector.
</Note>
