## Prerequisites

<div className="dynamo-prereq-grid">
<div><span>Hardware</span><strong>16x H100/H200 (Hopper) or B100/B200 (Blackwell); ~1.3 TB total GPU VRAM</strong></div>
<div><span>Runtime</span><strong>TensorRT-LLM (PyTorch backend)</strong></div>
<div><span>Traffic</span><strong>4K ISL / 200 OSL synthetic</strong></div>
<div><span>Concurrency</span><strong>32 (2 per GPU x 16 GPUs)</strong></div>
<div><span>MoE backend</span><strong>CUTLASS (default) on Hopper; DEEPGEMM on Blackwell</strong></div>
<div><span>Setup assets</span><strong>model-cache/model-cache.yaml and model-cache/model-download.yaml</strong></div>
</div>

<Warning>
Edit namespace, storage class, image tags, node selectors, resource claims, Hugging Face secrets, and cluster-specific placement before applying these manifests. Pick the variant that matches your GPU architecture — the Hopper and Blackwell manifests are not interchangeable.
</Warning>

## Deployment Instructions

<div id="deploy"></div>

Create the Hugging Face token secret and prepare the model cache once, then deploy the variant matching your hardware. Model download can take 30-60 minutes.

```bash
export NAMESPACE=your-namespace

kubectl create secret generic hf-token-secret \
  --from-literal=HF_TOKEN="your-token" \
  -n ${NAMESPACE}

# Edit storageClassName in model-cache/model-cache.yaml to match your cluster first.
kubectl apply -f recipes/qwen3-235b-a22b-fp8/model-cache/ -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=3600s
```

<h3 id="deploy-aggregated-hopper">Aggregated Hopper</h3>

4x TP4 x EP4 workers with KV-aware routing; default CUTLASS MoE backend.

```bash
kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/agg/hopper/deploy.yaml -n ${NAMESPACE}
```

<h3 id="deploy-aggregated-blackwell">Aggregated Blackwell</h3>

Same topology as the Hopper variant, with `moe_config.backend: DEEPGEMM` set in the engine config (required on SM100+).

```bash
kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/agg/blackwell/deploy.yaml -n ${NAMESPACE}
```

<h3 id="deploy-disaggregated-hopper">Disaggregated Hopper</h3>

6x prefill workers (TP2) and 1x decode worker (TP4 x EP4); default CUTLASS MoE backend.

```bash
kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/disagg/hopper/deploy.yaml -n ${NAMESPACE}
```

<h3 id="deploy-disaggregated-blackwell">Disaggregated Blackwell</h3>

Same P/D topology as the Hopper variant, with `moe_config.backend: DEEPGEMM` set for both prefill and decode configs.

```bash
kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/disagg/blackwell/deploy.yaml -n ${NAMESPACE}
```

## Smoke Test

<div id="smoke-test"></div>

Run a small request before starting the benchmark. Use the frontend service that matches your target.

```bash
export NAMESPACE=your-namespace

# Aggregated (either architecture):
kubectl port-forward svc/qwen3-235b-a22b-agg-frontend 8000:8000 -n ${NAMESPACE}
# Disaggregated (either architecture):

```

## Benchmark Instructions

<div id="benchmark"></div>

All four targets ship a `perf.yaml` running the same AIPerf job — 4K ISL / 200 OSL at concurrency 32 (2 per GPU x 16 GPUs) with `--request-count 320` — so aggregated vs disaggregated results are comparable within the same hardware architecture. Artifacts land on the `model-cache` PVC under `/model-cache/perf`.

```bash
aiperf profile \
  --model Qwen/Qwen3-235B-A22B-FP8 --tokenizer Qwen/Qwen3-235B-A22B-FP8 \
  --endpoint-type chat --endpoint /v1/chat/completions \
  --url http://qwen3-235b-a22b-agg-frontend:8000 --streaming \
  --synthetic-input-tokens-mean 4000 --synthetic-input-tokens-stddev 0 \
  --output-tokens-mean 200 --output-tokens-stddev 0 \
  --extra-inputs max_tokens:200 --extra-inputs min_tokens:200 --extra-inputs ignore_eos:true \
  --concurrency 32 --request-count 320 --warmup-request-count 32 \
  --random-seed 100
```

For the disaggregated targets the job points at `http://qwen3-235b-a22b-disagg-frontend:8000` instead. Apply the manifest matching your deployed target:

```bash
kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/agg/hopper/perf.yaml -n ${NAMESPACE}
# OR: kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/agg/blackwell/perf.yaml -n ${NAMESPACE}
# OR: kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/disagg/hopper/perf.yaml -n ${NAMESPACE}
# OR: kubectl apply -f recipes/qwen3-235b-a22b-fp8/trtllm/disagg/blackwell/perf.yaml -n ${NAMESPACE}
```

## Notes

- The Hopper/Blackwell split is required with TRT-LLM 1.3.x: the default CUTLASS MoE backend falls through to a Hopper-specific JIT path on SM100 and crashes, so Blackwell needs `moe_config.backend: DEEPGEMM`. DEEPGEMM in turn crashes on Hopper due to a scale-factor dtype mismatch — hence two separate variants per topology.
- Chunked prefill is enabled for the aggregated targets and disabled for the disaggregated targets.
- All targets use KV-aware routing (`--router-mode kv`) at the frontend.
- Model download may take 30-60 minutes; update `storageClassName` in `model-cache/model-cache.yaml` before deploying.
