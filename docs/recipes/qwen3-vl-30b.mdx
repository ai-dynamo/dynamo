## When to Use

Use this recipe when your multimodal traffic repeatedly references the same or similar images. The
embedding cache stores multimodal embeddings so repeat image payloads can avoid recomputing the
vision path. Keep the cache-off run as the baseline before changing cache capacity, request rate, or
image reuse.

## Dataset

`data-gen/generate-datasets-job.yaml` creates 1,000 requests with 1 image per request, 400 user-text
tokens, and an image pool of 200. The first 200 requests use unique images; the remaining requests
reuse images from that pool.

```bash
export NAMESPACE=your-namespace

kubectl apply -f recipes/qwen3-vl-30b/data-gen/generate-datasets-job.yaml \
  -n ${NAMESPACE}

kubectl wait --for=condition=Complete job/qwen3-vl-30b-generate-datasets \
  -n ${NAMESPACE} \
  --timeout=3600s

kubectl logs job/qwen3-vl-30b-generate-datasets -n ${NAMESPACE}
```

Exact cache hit rates are not directly controlled by the dataset because cache eviction depends on
runtime behavior. Smaller image pools and larger cache capacity increase the chance that reused
images remain in cache.

## Configuration Notes

<Accordion title="Embedding cache">
The recipe uses vLLM's native multimodal embedding cache path. The deployment enables multimodal
serving, prefix caching, `DYN_VLLM_EMBEDDING_TRANSFER_MODE=nixl-write`, and
`--multimodal-embedding-cache-capacity-gb "${DYN_MULTIMODAL_EMBEDDING_CACHE_GB}"`.
</Accordion>

<Accordion title="Cache on/off comparison">
The checked-in `deploy.yaml` defaults to cache ON with `DYN_MULTIMODAL_EMBEDDING_CACHE_GB=10`.
For cache OFF, use `run-benchmark.sh off` or set the deployment env var to `0` and the benchmark
`CACHE_MODE` to `cache_off` before applying the manifests.
</Accordion>

<Accordion title="Dataset generation">
The dataset generator writes `qwen3_vl_1000req_1img_pool200.jsonl` under `/perf-cache/datasets`.
It uses HTTP image mode and a 200-image pool, so the deployment permits the benchmark image URLs with
`DYN_MM_ALLOW_INTERNAL=1`.
</Accordion>
