This recipe deploys Qwen3-VL-30B aggregated on a single GB200 with a 10 GiB multimodal embedding cache, delivering +16% output throughput and -28% average TTFT versus cache-off on repeated-image traffic. The cache-off run is a benchmark baseline, not a separate recipe target.

## Prerequisites

- A Kubernetes cluster with the Dynamo platform (operator) installed and **1x GB200** available.
- A Hugging Face token (the model downloads via the `hf-token-secret`).
- The `nvcr.io/nvidia/ai-dynamo/vllm-runtime:1.2.0` image (already pinned in `deploy.yaml`); the embedding cache uses vLLM 0.17+'s native `ec_both` ECConnector.

Create the namespace and token secret:

```bash
export NAMESPACE=your-namespace
kubectl create namespace ${NAMESPACE}
kubectl create secret generic hf-token-secret \
  --from-literal=HF_TOKEN="your-token" \
  -n ${NAMESPACE}
```

<Warning>
Edit cluster-specific values before applying these manifests: `storageClassName: "your-storage-class-name"` in `model-cache/model-cache.yaml` and your namespace.
</Warning>

## Deploy

Create storage, download the model, generate the dataset, then deploy. The dataset generation job is part of the reproduce path — `perf.yaml` fails fast if the generated JSONL is missing.

```bash
# 1. Create the model-cache, compilation-cache, and perf-cache PVCs.
kubectl apply -f recipes/qwen3-vl-30b/model-cache/model-cache.yaml -n ${NAMESPACE}

# 2. Download the model into the model-cache PVC.
kubectl apply -f recipes/qwen3-vl-30b/model-cache/model-download.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=3600s

# 3. Generate the multimodal dataset (1,000 requests, 1 image each, 200-image pool).
kubectl apply -f recipes/qwen3-vl-30b/data-gen/generate-datasets-job.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/qwen3-vl-30b-generate-datasets -n ${NAMESPACE} --timeout=3600s

# 4. Deploy. deploy.yaml defaults to cache ON (DYN_MULTIMODAL_EMBEDDING_CACHE_GB=10).
kubectl apply -f recipes/qwen3-vl-30b/vllm/agg-embedding-cache/deploy.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Ready dynamographdeployment/qwen3-vl-agg -n ${NAMESPACE} --timeout=900s
```

## Smoke Test

Run a small request before starting the benchmark. The Dynamo operator creates the `qwen3-vl-agg-frontend` service for this deployment.

```bash
kubectl port-forward svc/qwen3-vl-agg-frontend 8000:8000 -n ${NAMESPACE}

curl http://localhost:8000/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"Qwen/Qwen3-VL-30B-A3B-Instruct-FP8","messages":[{"role":"user","content":"Write a one-sentence readiness check."}],"max_tokens":64}'
```

## Benchmark

The checked-in [`perf.yaml`](https://github.com/ai-dynamo/dynamo/blob/main/recipes/qwen3-vl-30b/vllm/agg-embedding-cache/perf.yaml) wraps this AIPerf run (aiperf 0.6.0):

```bash
aiperf profile --model Qwen/Qwen3-VL-30B-A3B-Instruct-FP8 --input-file qwen3_vl_1000req_1img_pool200.jsonl \
  --custom-dataset-type single_turn --url http://qwen3-vl-agg-frontend:8000 --streaming \
  --request-count 1000 --concurrency 64 --request-rate-mode constant --warmup-request-count 3 \
  --extra-inputs max_tokens:150 --extra-inputs min_tokens:150 --extra-inputs ignore_eos:true
```

```bash
kubectl apply -f recipes/qwen3-vl-30b/vllm/agg-embedding-cache/perf.yaml -n ${NAMESPACE}

# Follow benchmark logs in real time
kubectl logs -f qwen3-vl-agg-benchmark -n ${NAMESPACE}
```

Artifacts land in `/perf-cache/artifacts/qwen3_vl_30b_embedding_cache/agg/<cache_mode>` on the `perf-cache` PVC. The helper script [`run-benchmark.sh`](https://github.com/ai-dynamo/dynamo/blob/main/recipes/qwen3-vl-30b/vllm/agg-embedding-cache/run-benchmark.sh) automates cache-on/cache-off runs.

## Expected Performance

Measured on a single aggregated GB200 replica with the vLLM backend; cache OFF is the benchmark baseline (see the related Feature Benchmark for the full A/B):

| Metric | Cache ON | Cache OFF | Delta |
| --- | ---: | ---: | ---: |
| Output TPS | 3,575.6 tok/s | 3,072.3 tok/s | +16.4% |
| TTFT avg | 526.0 ms | 727.5 ms | -27.7% |
| TTFT p50 | 356.8 ms | 510.8 ms | -30.1% |
| ITL avg | 14.1 ms | 15.5 ms | -8.8% |
| Request latency avg | 2,630.0 ms | 3,035.7 ms | -13.4% |

Enabling the embedding cache shows an average improvement of +16% throughput, -28% TTFT, and -13% request latency.

## Notes

- The embedding cache uses vLLM's native `ec_both` ECConnector role, supported in vLLM 0.17+; no patches required. See [multimodal-vllm.md](https://github.com/ai-dynamo/dynamo/blob/main/docs/features/multimodal/multimodal-vllm.md#embedding-cache).
- Run the cache-off experiment from the related Feature Benchmark page by setting `DYN_MULTIMODAL_EMBEDDING_CACHE_GB=0` in `deploy.yaml` and `CACHE_MODE=cache_off` in `perf.yaml`; do not present it as a separate recipe choice.
- Exact cache hit rates cannot be explicitly controlled via the dataset because of LRU eviction; shrinking the image pool relative to request count, or growing the cache capacity, raises the hit probability.
- Each generated request carries 400 tokens of user text; the first 200 of 1,000 requests use unique images, the remaining 800 repeat images from the pool.
