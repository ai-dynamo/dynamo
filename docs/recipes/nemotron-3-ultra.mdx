## Prerequisites

<div className="dynamo-prereq-grid">
<div><span>Hardware</span><strong>4x B200 (TP4) or 8x H200 (TP8); 8x B200 for the disaggregated target</strong></div>
<div><span>Runtime</span><strong>nvcr.io/nvidia/ai-dynamo/vllm-runtime:1.3.0-nemotron-ultra-dev.1</strong></div>
<div><span>Dataset</span><strong>Bundled Moontrace JSONL traces (15%, 30%, full)</strong></div>
<div><span>Traffic</span><strong>Chat 8K/1K and agentic 64K/400 Mooncake-format replays</strong></div>
<div><span>Secrets</span><strong>NGC pull secret nvcr-secret; HF token secret hf-token-secret</strong></div>
<div><span>Setup assets</span><strong>model-cache/model-cache.yaml, model-download.yaml, model-validate.yaml</strong></div>
</div>

You need the Dynamo Platform installed on the target cluster with the DGD CRDs served, plus a `shared-model-cache` PVC containing the tokenizer-patched Ultra model view (or permission to create and populate it with the manifests in `model-cache/`).

<Warning>
Edit namespace, storage class, image tags, node selectors, and cluster-specific placement before applying these manifests.
</Warning>

## Deployment Instructions

<div id="deploy"></div>

First create and populate the model cache, then validate the patched model view before deploying any server:

```bash
export NAMESPACE=your-namespace

# Create + populate the shared model cache (edit storage class first if needed)
kubectl apply -f recipes/nemotron-3-ultra/model-cache/model-cache.yaml -n ${NAMESPACE}
kubectl apply -f recipes/nemotron-3-ultra/model-cache/model-download.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/nemotron-ultra-model-download -n ${NAMESPACE} --timeout=12h

# Validate the patched model view (model/tokenizer/parser files) before deploying
kubectl apply -f recipes/nemotron-3-ultra/model-cache/model-validate.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/nemotron-ultra-model-validate -n ${NAMESPACE} --timeout=30m
```

### Aggregated targets (B200/H200 × chat/agentic)

Pick the SKU and use case, then apply the matching manifest. The DGD is named `ultra-agg-<sku>-<usecase>-mtp`:

```bash
SKU=b200       # or h200
USECASE=chat   # or agentic

kubectl apply -f recipes/nemotron-3-ultra/vllm/agg-${SKU}-${USECASE}-mtp/deploy.yaml -n ${NAMESPACE}

DGD=ultra-agg-${SKU}-${USECASE}-mtp
kubectl get dgd ${DGD} -n ${NAMESPACE} -w
```

No-MTP fallback manifests live alongside the MTP ones at `vllm/agg-<sku>-<usecase>-nomtp/deploy.yaml`; their DGD names carry the `-nomtp` suffix.

### B200 disaggregated agentic (1P1D)

The disaggregated recipe is currently agentic B200, no-MTP only. The DGD is named `ultra-disagg-b200-1p1d-agentic-nomtp`:

```bash
kubectl apply -f recipes/nemotron-3-ultra/vllm/disagg-b200-agentic/deploy.yaml -n ${NAMESPACE}
kubectl get dgd ultra-disagg-b200-1p1d-agentic-nomtp -n ${NAMESPACE} -w
```

## Smoke Test

<div id="smoke-test"></div>

Each DGD exposes a frontend service named `<dgd-name>-frontend` (for example `ultra-agg-b200-chat-mtp-frontend`):

```bash
export NAMESPACE=your-namespace
DGD=ultra-agg-b200-chat-mtp   # match the target you deployed
kubectl port-forward svc/${DGD}-frontend 8000:8000 -n ${NAMESPACE}

MODEL_ID=nvidia/NVIDIA-Nemotron-3-Ultra-550B-A55B-NVFP4

curl http://localhost:8000/v1/models
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d "{\"model\":\"${MODEL_ID}\",
       \"messages\":[{\"role\":\"user\",\"content\":\"Hello!\"}],
       \"max_tokens\":64,
       \"chat_template_kwargs\":{\"enable_thinking\":false,\"force_nonempty_content\":true}}"
```

## Benchmark Instructions

<div id="benchmark"></div>

A single AIPerf trace-replay Job, `perf/perf.yaml`, covers every target. Edit `ENDPOINT`, `TRACE_FILE`, and `CONCURRENCY` in the Job env block to match the variant you deployed (defaults target B200 chat MTP at concurrency 18). The Job runs this AIPerf raw Moontrace replay:

```bash
aiperf profile \
  -m nvidia/NVIDIA-Nemotron-3-Ultra-550B-A55B-NVFP4 \
  --tokenizer /opt/models/patched/NVIDIA-Nemotron-3-Ultra-550B-A55B-NVFP4 \
  --tokenizer-trust-remote-code \
  --input-file /opt/models/traces/nim_turbo_8k_1k_70kv_chat_new_noschedule_short_15perc.jsonl \
  --custom-dataset-type mooncake_trace \
  --prompt-input-tokens-block-size 512 \
  --url http://ultra-agg-b200-chat-mtp-frontend:8000 \
  --streaming --use-server-token-count \
  --extra-inputs ignore_eos:true \
  --concurrency 18 --random-seed 42
```

Stage the bundled traces on the `shared-model-cache` PVC under `/opt/models/traces/` (see the [perf README](https://github.com/ai-dynamo/dynamo/blob/main/recipes/nemotron-3-ultra/perf/README.md) for the helper-pod workflow), then run the Job:

```bash
kubectl apply -f recipes/nemotron-3-ultra/perf/perf.yaml -n ${NAMESPACE}
kubectl logs -n ${NAMESPACE} -l job-name=ultra-bench -f
kubectl wait --for=condition=Complete job/ultra-bench -n ${NAMESPACE} --timeout=7200s
```

Artifacts land on the PVC under `/opt/models/perf/<epoch>_ultra-bench/`. For concurrency sweeps, delete the worker pods between runs so residual KV/prefix-cache state does not skew results.

## Expected Performance

B200 rows use 15% raw Moontrace replay with `raw_direct_no_filter` trace semantics; H200 rows use 300-sample replay evidence. `User output tok/s` is Gen TPS/user p50 from AIPerf; `System output tok/s/GPU` is TPS/GPU.

| Recipe | GPU | Topology | Workload | MTP | Concurrency | User output tok/s | System output tok/s/GPU |
|--------|-----|----------|----------|-----|-------------|-------------------|-------------------------|
| `vllm/agg-b200-agentic-mtp/deploy.yaml` | B200 | AGG | agentic | yes | 20 | 80.6 | 310.8 |
| `vllm/agg-b200-agentic-nomtp/deploy.yaml` | B200 | AGG | agentic | no | 8 | 99.5 | 175.9 |
| `vllm/agg-b200-chat-mtp/deploy.yaml` | B200 | AGG | chat | yes | 18 | 52.0 | 201.4 |
| `vllm/agg-b200-chat-nomtp/deploy.yaml` | B200 | AGG | chat | no | 16 | 51.0 | 181.3 |
| `vllm/disagg-b200-agentic/deploy.yaml` | B200 | 1P1D | agentic | no | 32 | 61.6 | 231.1 |
| `vllm/agg-h200-agentic-mtp/deploy.yaml` | H200 | AGG | agentic | yes | 8 | 53.2 | 27.4 |
| `vllm/agg-h200-agentic-nomtp/deploy.yaml` | H200 | AGG | agentic | no | 8 | 52.3 | 26.5 |
| `vllm/agg-h200-chat-mtp/deploy.yaml` | H200 | AGG | chat | yes | 10 | 58.7 | 46.8 |
| `vllm/agg-h200-chat-nomtp/deploy.yaml` | H200 | AGG | chat | no | 8 | 54.2 | 43.0 |

Treat each row together with its matching recipe, image, trace, and server-shape artifacts.

## Notes

- This is a Day-0 recipe on a dedicated dev runtime image (`vllm-runtime:1.3.0-nemotron-ultra-dev.1`); it is functional and benchmarked but not yet promoted to a release runtime image.
- The recipes pin `VLLM_DISABLED_KERNELS=FlashInferFP8ScaledMMLinearKernel` and pass `--no-enable-flashinfer-autotune` on vLLM workers. Do not remove these unless rerunning the benchmark qualification — they select the non-FlashInfer FP8 linear kernel path and avoid a measured vLLM 0.22 FlashInfer FP8 regression.
- No-MTP fallback manifests (`vllm/agg-*-nomtp/deploy.yaml`) are included for every aggregated target; their measured rows appear in the performance table above.
- Reasoning is controlled per request via `chat_template_kwargs` (`enable_thinking`, `force_nonempty_content`) and `nvext.max_thinking_tokens`. Do not send `force_nonempty_content` as a top-level request parameter. Top-level reasoning controls such as `include_reasoning` and `reasoning_effort` are part of shared Dynamo API compatibility work, not Ultra-specific failures.
- Raw Moontrace replay may contain over-context or pathological long-generation rows. Preserve them as HTTP/error evidence rather than dropping them silently.
- Tool calling uses the `qwen3_coder` parser; reasoning parsing uses the model-local `ultra_v3_reasoning_parser.py` (validated by the model-validate Job).
