## When to Use

Use this recipe when the model is `nvidia/DeepSeek-V3.2-NVFP4`, the target hardware is GB200 NVL72,
and the traffic has long coding prompts with repeated context. The disaggregated variant is the
candidate serving path; the aggregated round-robin variant is the control run for measuring the
effect of KV-aware routing, prefill/decode separation, and WideEP.

## Workload

The benchmark trace is synthesized from the Mooncake conversation trace by increasing prompt length
and prefix reuse. Use Dynamo's prefix data generator to reproduce the trace shape:

```bash
datagen synthesize \
  --input-file conversation_trace.jsonl \
  --prefix-len-multiplier 16 \
  --prompt-len-multiplier 10 \
  --max-isl 110000 \
  --num-requests 10000
```

The resulting trace has 10,000 requests, 39,186 average input tokens, 344 average output tokens,
about 44% block/token KV hit rate, and about 57% shared-prefix input tokens.

## Configuration Notes

<Accordion title="Disagg + KV router + WideEP">
The frontend runs in KV-router mode. Prefill and decode workers use TensorRT-LLM with DEP8,
64-token KV blocks, UCX cache transfer, FP8 KV cache, and a 121K max sequence length. Decode uses
`moe_config.backend: WIDEEP`; prefill uses the TRT-LLM MoE backend. Both worker groups use
ComputeDomain resource claims for MNNVL placement.
</Accordion>

<Accordion title="Aggregated round-robin">
The aggregate baseline keeps the NVIDIA NVFP4 checkpoint, DEP8, 64-token KV blocks, MNNVL allreduce,
FP8 KV cache, and 121K max sequence length, but uses four aggregate workers with round-robin routing.
</Accordion>

<Accordion title="Benchmark job">
The benchmark job installs `aiperf==0.6.0` and pins `transformers==4.57.6` so the DeepSeek V3.2
tokenizer loads correctly. It runs a short synthetic warmup before the fixed-schedule trace replay.
</Accordion>
