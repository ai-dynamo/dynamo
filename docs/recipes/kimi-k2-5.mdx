## When to Use

Use this page when your deployment decision starts with `nvidia/Kimi-K2.5-NVFP4` and you need to understand which checked-in recipe asset matches your backend, hardware, and workload shape. The variant role column is part of the design: a control row, treatment row, alternative row, and deploy-only row should not be interpreted the same way.

## Results

The source README includes expected metrics for the four benchmarked TensorRT-LLM arms. TokenSpeed is intentionally excluded from the comparison because it has deploy-only evidence.

| Variant | tok/s/user avg | tok/s/GPU | Decision signal |
| --- | ---: | ---: | --- |
| Disagg + Eagle + KV router + offload | ~130 tok/s/user avg | ~5,400 tok/s/GPU | Best throughput/interactivity |
| Agg + Eagle + KV router | ~85 tok/s/user avg | ~4,400 tok/s/GPU | Keeps routing and Eagle |
| Agg + Eagle + round-robin | ~95 tok/s/user avg | ~4,000 tok/s/GPU | Isolates routing |
| Agg + round-robin | ~105 tok/s/user avg | ~1,700 tok/s/GPU | Base control |

## Deploy

<h3 id="deploy-disagg--eagle--kv-router--offload">Deploy: Disagg + Eagle + KV router + offload</h3>

Best benchmarked point for throughput and interactivity on KV-reuse-heavy traffic.

```bash
export NAMESPACE=your-namespace


# Prepare shared storage and model assets using the setup manifests in `recipes/kimi-k2.5`.
kubectl apply -f recipes/kimi-k2.5/trtllm/disagg-eagle-kv-router/deploy.yaml -n ${NAMESPACE}
# Run the benchmark for this variant.
kubectl apply -f recipes/kimi-k2.5/trtllm/disagg-eagle-kv-router/perf.yaml -n ${NAMESPACE}
```

<h3 id="deploy-agg--eagle--kv-router">Deploy: Agg + Eagle + KV router</h3>

Keeps prefix-aware routing and Eagle while removing P/D split and KV offload.

```bash
export NAMESPACE=your-namespace


# Prepare shared storage and model assets using the setup manifests in `recipes/kimi-k2.5`.
kubectl apply -f recipes/kimi-k2.5/trtllm/agg-eagle-kv-router/deploy.yaml -n ${NAMESPACE}
# Run the benchmark for this variant.
kubectl apply -f recipes/kimi-k2.5/trtllm/agg-eagle-kv-router/perf.yaml -n ${NAMESPACE}
```

<h3 id="deploy-agg--eagle--round-robin">Deploy: Agg + Eagle + round-robin</h3>

Keeps Eagle and switches routing from KV-aware to round-robin.

```bash
export NAMESPACE=your-namespace


# Prepare shared storage and model assets using the setup manifests in `recipes/kimi-k2.5`.
kubectl apply -f recipes/kimi-k2.5/trtllm/agg-eagle-round-robin/deploy.yaml -n ${NAMESPACE}
# Run the benchmark for this variant.
kubectl apply -f recipes/kimi-k2.5/trtllm/agg-eagle-round-robin/perf.yaml -n ${NAMESPACE}
```

<h3 id="deploy-agg--round-robin">Deploy: Agg + round-robin</h3>

Base control without Eagle; compare against Eagle rows before choosing speculation.


- The four TensorRT-LLM arms should be run one at a time because the manifests are sized for the full 24 GPUs.
