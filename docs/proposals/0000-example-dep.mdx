## Summary

Dynamo Enhancement Proposals (DEPs) can be authored as markdown in this repository under `docs/proposals/` and rendered on a dedicated Proposals tab at docs.nvidia.com/dynamo. Each rendered page carries a status banner, a metadata card, and a read-only mirror of the DEP's GitHub comments (inline PR review, PR conversation, and the tracking-issue thread) so the design content, its review, and the open-ended debate live close to the code.

## Motivation

DEP content, line-level review, and open-ended debate have historically been split across a separate repository, pull request threads, and meetings. That split makes it hard for a reader to find the current state of a proposal and hard for a community member without repo access to weigh in.

Colocating the DEP markdown with the code, rendering it on the docs site, and mirroring the tracking-issue and PR comments onto the page keeps all three concerns in one discoverable place while reusing infrastructure that already exists: GitHub issues, pull requests, and the Fern docs site. There is no new comment system to run — the page reads GitHub's public API and deep-links back to GitHub for every reply.

### Goals

* Keep DEP content next to the code it describes, versioned with the code.
* Render every DEP on the public docs site with an unambiguous status.
* Surface the DEP's GitHub discussion on the page and let any GitHub user join it (on the issue or PR) without repo write access.

### Non Goals

* Replacing the tracking issue as the anchor and durable debate thread for a DEP.
* Accepting comments on the docs page itself. Authoring and replies stay on GitHub; the page is a read-only mirror.
* Migrating the existing proposals from `ai-dynamo/enhancements`. That is a separate decision.

### Requirements

#### REQ 1: Status Must Be Unambiguous

Every rendered DEP page **MUST** show its status (Draft, Under Review, Accepted, Rejected, Deferred, Implemented, or Replaced) prominently at the top, so an in-flight proposal never reads as ratified.

#### REQ 2: Debate Must Be Open to the Community

Joining a DEP's discussion **MUST NOT** require write access to the repository. Any authenticated GitHub user **SHOULD** be able to comment on the linked issue or PR, and the page **SHOULD** surface those comments and deep-link to them.

## Proposal

A DEP is a markdown file under `docs/proposals/`, authored from `TEMPLATE.mdx`. It is wired into the Fern navigation under a Proposals tab. Two GitHub objects back each DEP:

* The **tracking issue** is the stable anchor and the home for durable, open-ended **design discussion**.
* The **pull request** that adds or edits the markdown is the **revision**: reviewers leave line-level comments on the exact text, plus general conversation on the PR.

The rendered page then shows:

1. A status banner, using a GitHub-style admonition that Fern converts to a native callout, plus a metadata card (status pill, fields, and action links).
2. A read-only mirror of the PR's inline review comments, anchored as highlights on the exact quoted text with expandable cards.
3. Two read-only discussion threads below the content — "Revision discussion" (PR conversation) and "Design discussion" (tracking-issue thread) — each with a deep-link button to reply on GitHub.

The lifecycle is: open a tracking issue (design discussion), author the DEP as a draft PR (line-level review + revision discussion), merge as Draft, and advance the status through follow-up PRs. Debate continues on the issue; the page always reflects it.

## Implementation Details

The Proposals tab is declared in `docs/index.yml`. The status banner uses `> [!WARNING]` for Draft, which the docs pipeline converts to a Fern callout. The metadata card is the `DepMetadata` component (`fern/components/DepMetadata.tsx`). The comment mirror is the `PrInlineComments` component (`fern/components/PrInlineComments.tsx`), which renders a `<div id="dep-pr-comments" data-pr=… data-issue=…>` mount; the client runtime in `fern/js/dep-pr-comments.js` (injected site-wide via `docs.yml` `js:`) reads those attributes and fetches three sources from GitHub's public REST API — `pulls/{pr}/comments` (inline review), `issues/{pr}/comments` (PR conversation), and `issues/{issue}/comments` (tracking issue) — then anchors the inline comments and renders the two discussion threads. It is read-only and unauthenticated; every card deep-links back to GitHub to reply.

### Deferred to Implementation

* Baking the mirrored comments in at build time for production, to avoid GitHub's unauthenticated 60-requests/hour/IP rate limit on high-traffic pages.

## Alternate Solutions

### Alt 1: Keep All DEPs in ai-dynamo/enhancements Only

**Pros**:

* One home; no duplication of process.

**Cons**:

* Content is not rendered on the docs site and is not colocated with the code.
* Community debate is limited to those who watch the enhancements repo.

**Reason Rejected**:

* Does not meet REQ 1 (rendered, unambiguous status) or REQ 2 (the discussion surfaced on the page and open to the community).

## Background

Enhancement-proposal processes such as Kubernetes KEPs and Rust RFCs store proposals as markdown in the repository and review them through pull requests. This example follows that lineage and adds a rendered docs page plus a read-only mirror of the DEP's GitHub issue and PR comments.
