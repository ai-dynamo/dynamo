A Dynamo Enhancement Proposal (DEP) captures an architecture, design, or process decision along with the motivation behind it. DEPs are owned by a SIG (Special Interest Group) and live in the dedicated `ai-dynamo/enhancements` repository. This Proposals tab renders them on docs.nvidia.com so the design content, its line-level review, and its discussion stay visible alongside the rest of the Dynamo docs.

Dynamo is multi-product and multi-repo, and is adopting SIGs. Enhancement proposals are cross-cutting, so they belong in a dedicated repository owned by the relevant SIG rather than inside any one code repo. This follows the Kubernetes model, where SIGs own KEPs in `kubernetes/enhancements` while implementation lands across many repos, and the same dedicated-repo pattern used by [Rust](https://github.com/rust-lang/rfcs), [React](https://github.com/reactjs/rfcs), [Swift](https://github.com/swiftlang/swift-evolution), and [Python](https://github.com/python/peps). The full process is defined in DEP-0001.

The docs site adds a visibility and discussion layer on top of that repository, reusing infrastructure that already exists: GitHub issues, pull requests, and this docs site. There is no separate comment service. The rendered page mirrors the DEP's GitHub comments read-only and deep-links back to GitHub for every reply.

> **Current State.** What renders today is a DEP page on this tab plus a read-only mirror of its GitHub review and discussion, with comments pulled cross-repo from the DEP's pull request and tracking issue (see the Nova page, which mirrors `ai-dynamo/enhancements` PR #61). ## The Model: Issue + PR, Mirrored on the Page

A DEP is backed by two GitHub objects and rendered as one page, each doing the job it is best at:

- **Pull request** (the revision): the PR in `ai-dynamo/enhancements` that adds or edits the DEP markdown. Reviewers leave **inline line-level comments** on the exact text, plus general **conversation** on the PR.
- **Tracking issue**: the stable anchor for the DEP's status and cross-references, and the home for durable, open-ended **design discussion**. Any GitHub user can join it without repository write access. A Linear issue (`DYN-####`) links implementation PRs across the code repos to the DEP, which GitHub's own same-repo issue linking cannot do across a multi-repo estate.
- **Content**: the DEP markdown in `ai-dynamo/enhancements`, owned by its SIG.
- **Rendered mirror**: the published page shows a read-only mirror of the comment surfaces: inline review comments anchored as highlights on the quoted text, a "Revision discussion" thread (PR conversation), and a "Design discussion" thread (the tracking issue). Each has a deep-link button to reply on GitHub.

## Workflow

1. **Open a tracking issue and identify the owning SIG.** The issue is the anchor and the durable design-discussion thread for the DEP.
2. **Author the DEP as a pull request in `ai-dynamo/enhancements`.** Start from the SIG's DEP template, fill in the required sections, and let reviewers use PR line comments and suggestions for precise, line-level review.
3. **Render it on this tab.** Add a page that points the comment mirror at the DEP's enhancements PR and tracking issue (set `pr` / `issue` on the `<PrInlineComments>` block), with the status banner set to its current state. Auto-rendering from the enhancements repo is planned; today the page is added here.
4. **Discuss on the tracking issue.** Open-ended debate lives on the tracking issue, so community members without repo access can participate; the rendered page mirrors it in the "Design discussion" section. Keep the issue linked from the metadata card.
5. **Advance the status via follow-up PRs.** As the proposal moves through Under Review to Accepted (or to Rejected, Deferred, or Replaced), update the status banner and header. Significant changes after acceptance get a new DEP; the original is marked Replaced.

## Status Banner Convention

Every rendered DEP page opens with a status banner so an in-flight proposal never reads as ratified. Use a GitHub-style admonition, which this docs pipeline converts to a native callout automatically (see `fern/convert_callouts.py`). Set the banner to match the DEP's current status:

| Status | Source admonition | Renders as |
| :----- | :---------------- | :--------- |
| Draft | `> [!WARNING]` | Warning |
| Under Review (a.k.a. Proposed) | `> [!IMPORTANT]` | Info |
| Accepted (a.k.a. Approved) | `> [!NOTE]` | Note |
| Rejected, Deferred, or Replaced (a.k.a. Superseded) | `> [!CAUTION]` | Error |

Example banner for a Draft proposal:

```md
> [!WARNING]
> **Status: Draft.** This proposal is under active discussion and is not an accepted decision.
```

Keep the `**Status**` header field and the banner in sync. The header field carries the full enum value; the banner carries the reader-facing callout.

## Commenting on a DEP

Comments are hosted by GitHub, not by the docs site. The rendered page is a read-only mirror; every reply happens on GitHub, so nothing new needs to be installed or run. The `PrInlineComments` component (`fern/components/PrInlineComments.tsx`) renders a mount div, and the client runtime (`fern/js/dep-pr-comments.js`, injected site-wide via `docs.yml` `js:`) reads its `data-owner` / `data-repo` / `data-pr` / `data-issue` attributes and fetches three sources from GitHub's public REST API:

- **Inline review** (`GET /repos/{owner}/{repo}/pulls/{pr}/comments`): line comments from the PR, anchored as highlights on the exact quoted text with expandable cards. Each links to that comment's reply UI on the PR.
- **Revision discussion** (`GET /repos/{owner}/{repo}/issues/{pr}/comments`): the PR's general conversation, as a threaded list. (PR conversation lives on the issues endpoint.)
- **Design discussion** (`GET /repos/{owner}/{repo}/issues/{issue}/comments`): the tracking-issue thread, the durable debate.

Automation accounts (GitHub Copilot and anything ending in `[bot]`) are filtered out of every surface so the page shows human debate. Each thread has a "Comment on the PR/issue" deep-link, and the design section shows an "open a tracking issue" prompt when no issue is linked. Set the sources per page, pointing `owner` / `repo` at the DEP's home repository (`ai-dynamo/enhancements`):

```mdx
<PrInlineComments owner="ai-dynamo" repo="enhancements" pr={12345} issue={67890} />
```

The fetch is unauthenticated, so it is subject to GitHub's 60-requests/hour/IP limit; on a 403 the affected section degrades to a notice plus a link to the thread. For high-traffic production pages, bake the comments in at build time to remove that limit.

## Relationship to ai-dynamo/enhancements

`ai-dynamo/enhancements` is the home for DEPs and holds the approved [`0000-dep-process`](https://github.com/ai-dynamo/enhancements/blob/main/deps/0000-dep-process.md). DEP-0001 defines the current process for the SIG model and adds this rendered discussion layer; whether it formalizes and extends `0000-dep-process` or supersedes it is being settled in review. This tab does not move DEPs out of that repository. It renders them and mirrors their review and discussion so the design stays visible next to the code.
