## Benchmark Question

How do aggregated, single-node disaggregated, and multi-node disaggregated vLLM topologies compare when normalized by GPU?

The three arms intentionally use different GPU counts (4, 8, and 16x H100/H200), so concurrency is scaled at 16 per GPU and results should be read as total throughput **and** TPS/GPU together. All three topologies are also deployable recipe targets, so this benchmark doubles as a sizing guide.

## Reproduce

<div id="reproduce"></div>

Each arm's `perf.yaml` computes total concurrency as 16 x GPU count and wraps this AIPerf command:

```bash
aiperf profile --model RedHatAI/Llama-3.3-70B-Instruct-FP8-dynamic \
  --endpoint-type chat --endpoint /v1/chat/completions \
  --url http://<frontend>:8000 --streaming \
  --synthetic-input-tokens-mean 8192 --synthetic-input-tokens-stddev 0 \
  --output-tokens-mean 1024 --output-tokens-stddev 0 \
  --extra-inputs max_tokens:1024 --extra-inputs min_tokens:1024 \
  --concurrency <16*gpu_count> --request-count <10*concurrency> \
  --warmup-request-count <concurrency>
```

The frontend services are `llama3-70b-agg-frontend`, `llama3-70b-disagg-sn-frontend`, and `llama3-70b-disagg-mn-frontend`. Deploy one arm at a time:

```bash
export NAMESPACE=your-namespace

# One-time prep: storage + model download (update storageClassName in model-cache.yaml first)
kubectl apply -f recipes/llama-3-70b/model-cache/ -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=3600s

# Pick one arm from the table above, deploy it, wait for readiness, then apply its perf.yaml.
kubectl apply -f recipes/llama-3-70b/vllm/<arm>/deploy.yaml -n ${NAMESPACE}
kubectl apply -f recipes/llama-3-70b/vllm/<arm>/perf.yaml -n ${NAMESPACE}
```

## Benchmark notes

- The source does not publish result numbers; run all three arms on your hardware and compare total output TPS alongside TPS/GPU, since GPU counts differ per arm.
- The model uses FP8 dynamic quantization applied at runtime; the download takes roughly 15-30 minutes.
- The `agg` and `disagg-single-node` arms also ship optional GAIE (Gateway API Inference Extension) manifests under their `gaie/` subfolders.
