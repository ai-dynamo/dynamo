## Benchmark Question

How much throughput and interactivity do we gain as KV routing, Eagle3 speculative decoding, prefill/decode disaggregation, and KV cache offload are composed?

All four arms run Dynamo + TensorRT-LLM on **6x GB200 nodes (24 GPUs total)** with MNNVL. The baseline is plain aggregated round-robin serving; each step adds a feature, ending at the full disaggregated stack (3x DEP4 prefill + 3x TEP4 decode, KV-aware routing, Eagle3, TRTLLM-native KV offload).

## Results

<div id="results"></div>

The disaggregated configuration with KV-aware routing, Eagle3 decoding, and KV offloading achieves the best system throughput and interactivity:

| Arm | tok/s/user (avg) | tok/s/GPU |
| --- | ---: | ---: |
| Disagg + Eagle3 + KV routing + offload | ~130 | ~5,400 |
| Agg + Eagle3 + KV routing | ~85 | ~4,400 |
| Agg + Eagle3 + round-robin | ~95 | ~4,000 |
| Agg + round-robin (no Eagle3) | ~105 | ~1,700 |

The full disaggregated stack delivers roughly **3x the per-GPU throughput** of the plain aggregated baseline while also improving per-user token speed — it dominates the throughput-interactivity Pareto frontier in the source plot.

## Reproduce

<div id="reproduce"></div>

The trace emulates a long-context, KV-reuse-heavy agentic coding workload (~200k-token context window, multi-turn sessions with restart-splits and a layered prefix-cache model). Generate it following the [dataset instructions in the AIPerf repository](https://github.com/ai-dynamo/aiperf/blob/1ecc2eac988eedc0e3a79b4c2d1063bfc295a014/src/aiperf/dataset/agentic_code_gen/datasets/1k_sessions_200k_ctx/manifest.json), then copy it to `/model-cache/traces/agent_trace_data/dataset.jsonl` on the PVC.

Each arm's `perf.yaml` runs a warmup pass and then wraps this AIPerf command (concurrency 32 for the disaggregated arm, 24 for the aggregated Eagle3 arms, 8 for the baseline):

```bash
aiperf profile -m nvidia/Kimi-K2.5-NVFP4 \
  --tokenizer nvidia/Kimi-K2.5-NVFP4 --tokenizer-trust-remote-code \
  --input-file /model-cache/traces/agent_trace_data/dataset.jsonl \
  --custom-dataset-type mooncake_trace \
  --url http://<frontend>:8000 \
  --streaming --extra-inputs ignore_eos:true \
  --concurrency <8|24|32> --random-seed 42 \
  --benchmark-duration 3600 --concurrency-ramp-duration 60 \
  --goodput "time_to_first_token:5000 inter_token_latency:10"
```

Deploy one arm at a time — each is sized for the full 24 GPUs:

```bash
export NAMESPACE=your-namespace

# One-time prep: storage, ComputeDomain, model + Eagle3 head download
kubectl apply -f recipes/kimi-k2.5/model-cache/model-cache.yaml -n ${NAMESPACE}
kubectl apply -f recipes/kimi-k2.5/model-cache/compute-domain.yaml -n ${NAMESPACE}
kubectl apply -f recipes/kimi-k2.5/model-cache/model-download.yaml -n ${NAMESPACE}
kubectl wait --for=condition=Complete job/model-download -n ${NAMESPACE} --timeout=3600s

# Pick one arm from the table above, deploy it, wait for readiness, then apply its perf.yaml.
kubectl apply -f recipes/kimi-k2.5/trtllm/<arm>/deploy.yaml -n ${NAMESPACE}
kubectl apply -f recipes/kimi-k2.5/trtllm/<arm>/perf.yaml -n ${NAMESPACE}
```

## Benchmark notes

- The manifests ship with a placeholder image tag (`nvcr.io/nvidia/ai-dynamo/tensorrtllm-runtime:my-tag`) — set a Dynamo TRT-LLM runtime image (v1.1.1~) that supports Kimi-K2.5 + Eagle3 in each `deploy.yaml` before applying.
- Your HuggingFace token needs access to both `nvidia/Kimi-K2.5-NVFP4` and the `nvidia/Kimi-K2.5-Thinking-Eagle3` speculative-decoding head.
- If you rename the ComputeDomain CR, mirror the change in every `deploy.yaml` under `extraPodSpec.resourceClaims` and `resources.claims`.

## Winning Configuration

The disaggregated Eagle3 + KV router + offload arm is the winning configuration and is deployable from its arm assets above. A recommended Recipe may be promoted from this benchmark in a future release; the aggregated arms exist as benchmark steps and controls.
