title: Check Local Deployment Health
subtitle: Query frontend and worker health endpoints and enable active worker checks

Use Dynamo's HTTP health endpoints to check local processes or configure an external supervisor. The
frontend serves its endpoints on the HTTP port; workers serve them through the system-status server
when `DYN_SYSTEM_PORT` is enabled.

## Query Health Endpoints

<Steps toc={true}>
<Step title="Start Dynamo components" id="start-dynamo-components">

```bash
python -m dynamo.frontend &
DYN_SYSTEM_PORT=8081 python -m dynamo.vllm \
  --model Qwen/Qwen3-0.6B \
  --enforce-eager &
```

</Step>
<Step title="Check the frontend" id="check-the-frontend">

```bash
curl --silent --show-error http://localhost:8000/live | jq
curl --silent --show-error http://localhost:8000/health | jq
```

`/live` reports whether the HTTP service is running. `/health` reports the frontend status and the
worker endpoints and instances visible through discovery.

</Step>
<Step title="Check the worker" id="check-the-worker">

```bash
curl --silent --show-error http://localhost:8081/live | jq
curl --silent --show-error http://localhost:8081/health | jq
```

The worker returns `503 Service Unavailable` while its status is `notready` and `200 OK` after it
becomes `ready`. The worker's `/live` and `/health` paths currently report the same system status.

</Step>
</Steps>

For response fields, status codes, custom paths, and endpoint differences, see the
[Health Check Reference](../reference/observability/health-checks.mdx).

## Enable Active Worker Checks

Canary checks exercise an idle worker endpoint with a small inference request. Enable them on the
worker process when you need active failure detection:

```bash
export DYN_HEALTH_CHECK_ENABLED=true
export DYN_CANARY_WAIT_TIME=10
export DYN_HEALTH_CHECK_REQUEST_TIMEOUT=3

DYN_SYSTEM_PORT=8081 python -m dynamo.vllm \
  --model Qwen/Qwen3-0.6B \
  --enforce-eager
```

Normal successful traffic resets the idle timer. After the configured idle period, Dynamo sends a
canary request; a timeout marks the endpoint unhealthy. See
[Observability Architecture](../design-docs/observability.md#active-worker-health-checks) for the
lifecycle and [Environment Variables](../reference/observability/environment-variables.mdx#health-checks)
for configuration defaults.

## Troubleshoot Health Checks

- **Connection refused:** Set `DYN_SYSTEM_PORT` before starting a worker and query that port.
- **Worker remains `notready`:** Inspect worker startup logs and confirm the engine registered its
  serving endpoint.
- **Canary checks do not run:** Confirm `DYN_HEALTH_CHECK_ENABLED=true` is present in the worker's
  environment before startup.
- **Canary checks time out:** Inspect engine errors and increase `DYN_HEALTH_CHECK_REQUEST_TIMEOUT`
  only if normal inference latency justifies it.
