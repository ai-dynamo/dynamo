# Plan: Reject duplicate model names on the same endpoint

## Context

Currently, two different models with different names can register at the same endpoint (`namespace/component/endpoint`). This causes undefined behavior at the frontend, which expects a single model name per endpoint.

Multiple workers serving the same model must still be allowed. Those workers register separate `instance_id`s, but they share the same `display_name`.

The race between concurrent registrations must be handled. A simple pre-check is not enough by itself.

## Final implementation approach

Put the exclusivity logic in the public `Discovery::register()` trait method in `lib/runtime`, not in `lib/llm`.

That makes the guarded behavior the default registration path for all model registrations while preserving backend-specific raw registration code behind a private-ish trait hook named `register_internal()`.

The public flow is:

1. If the spec is not `DiscoverySpec::Model`, delegate directly to `register_internal()`.
2. If the spec is `DiscoverySpec::Model`, extract `display_name` from `card_json`.
3. Pre-check `DiscoveryQuery::EndpointModels { namespace, component, endpoint }`.
4. If an existing model at that endpoint has a different `display_name`, fail immediately.
5. Call `register_internal(spec)`.
6. Post-check the same endpoint again.
7. If a conflicting model now exists, call `unregister(instance.clone())` to roll back our own registration and return an error.

## Step 1: Update the `Discovery` trait in `lib/runtime/src/discovery/mod.rs`

Modify the discovery trait so:

- `register()` becomes a default async method on the trait.
- Introduce a new required method:

```rust
async fn register_internal(&self, spec: DiscoverySpec) -> Result<DiscoveryInstance>;
```

- `register()` contains the exclusivity logic for `DiscoverySpec::Model`.

Implementation details:

### Add small helper functions near the trait

Add:

```rust
fn extract_model_display_name(card_json: &serde_json::Value) -> Result<String>
fn find_conflicting_model_name(
    instances: &[DiscoveryInstance],
    requested_name: &str,
) -> Result<Option<String>>
```

Notes:

- Do not depend on `lib/llm` types from `lib/runtime`.
- Extract `display_name` directly from `card_json` via `serde_json::Value`.
- If `display_name` is missing or not a string, return an error.

### `register()` behavior

Inside `register()`:

- Match on `&spec`.
- For non-model specs, return `self.register_internal(spec).await`.
- For model specs, clone `namespace`, `component`, and `endpoint`, and extract `requested_name`.
- Build:

```rust
DiscoveryQuery::EndpointModels {
    namespace,
    component,
    endpoint,
}
```

- Pre-check `self.list(query.clone()).await?`.
- If any existing model has a different name, return:

```rust
"Cannot register model '{requested_name}' on endpoint '{namespace}/{component}/{endpoint}': a different model '{conflicting_name}' is already registered there"
```

- Call `self.register_internal(spec).await?`.
- Post-check `self.list(query).await?`.
- If a conflicting model appears after registration:
  - attempt `self.unregister(instance.clone()).await`
  - if rollback fails, wrap the same conflict error with:

```rust
format!(
    "failed to roll back conflicting model registration for instance {instance_id}: {unregister_err}",
    instance_id = instance.instance_id()
)
```

- Otherwise return the registered instance.

## Step 2: Rename backend implementations from `register` to `register_internal`

Update all `Discovery` impls to expose their raw backend registration logic through `register_internal()` instead of `register()`.

Files:

1. `lib/runtime/src/discovery/mock.rs`
2. `lib/runtime/src/discovery/kv_store.rs`
3. `lib/runtime/src/discovery/kube.rs`

Only the method name changes here. The internal behavior should remain the same.

## Step 3: Keep model registration call sites using plain `discovery.register(spec)`

Because the trait-level `register()` now contains the exclusivity logic, the call sites should keep using the normal API.

Files:

1. `lib/bindings/python/rust/lib.rs`
2. `lib/llm/src/local_model.rs`

Required state:

- Tensor/images/videos fast path in `lib/bindings/python/rust/lib.rs` calls:

```rust
discovery.register(spec).await.map_err(to_pyerr)?;
```

- Regular local-model attach path in `lib/llm/src/local_model.rs` calls:

```rust
let _instance = discovery.register(spec).await?;
```

No special llm-side wrapper should remain.

## Step 4: Add runtime-level unit tests in `lib/runtime/src/discovery/mock.rs`

Add focused tests against `MockDiscovery` so the trait behavior is validated directly.

Suggested helper:

```rust
fn model_spec(namespace: &str, component: &str, endpoint: &str, model_name: &str) -> DiscoverySpec
```

This helper should build a `DiscoverySpec::Model` using minimal JSON:

```rust
card_json: serde_json::json!({
    "display_name": model_name,
})
```

Add tests for:

1. Same model name, same endpoint -> both registrations succeed
2. Different model name, same endpoint -> second registration fails
3. Different model names, different endpoints -> both registrations succeed

For the rejection test:

- assert the error string contains:

```text
Cannot register model 'model-b' on endpoint 'ns/comp/generate': a different model 'model-a' is already registered there
```

- also assert only one model instance remains registered on that endpoint
