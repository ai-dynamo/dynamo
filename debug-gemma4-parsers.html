Gemma 4 tool/reasoning parser investigation

2026-07-09 · branch rmccormick/gemma4-parsers · source 485c298fff259ea228d4d86e23fe4153aead3c8a

Result: structured tool calls verified live on vLLM and SGLang. The checkpoint has a separate exact-copy limitation for the literal Gemma string-delimiter token.

Finding

Two Gemma 4 state machines disagree at an implicit tool boundary. vLLM 0.24.0 treats <|tool_call> as the end of reasoning and begins named-tool guided JSON. Dynamo's pinned Gemma reasoner waits only for <channel|>, so it classifies the marker and guided JSON as reasoning_content. The fix normalizes only the first structural <|tool_call> to a reasoning close for required/named tool choices. Later identical text is preserved as tool-argument data.

The branch wraps SGLang's spawned scheduler entrypoint and installs a narrow Gemma 4 compatibility shim in the child process. The shim makes token ID 48 (<|tool_call>) an alternate reasoning-end transition and preserves that state on request-local grammar copies. This is scoped to SGLang's Gemma4Detector.

Checkpoint-output limitation: the schema-valid SGLang result also fails character-exact SQL reproduction. It preserves the embedded tool marker but replaces the delimiter suffix and continues with explanatory prose inside the string. The parser correctly returns exactly the generated argument; it cannot reconstruct text the model did not emit.

Launch conclusions

  • Use all three requested parser selectors as gemma4 on both backends.
  • On SGLang 0.5.14, also use --constrained-json-disable-any-whitespace.
  • The Dynamo fix prevents structural tool syntax from leaking into reasoning and returns one named tool call on both engines.
  • Do not claim exact argument fidelity for this adversarial prompt; both backend runs show deterministic checkpoint/template corruption around <|"|>.
