# GPU Memory Service V1

GMS V1 is an experimental, Dynamo Snapshot-only owner for model Parameter
backing. It assumes one persistent GMS sidecar on the same physical GPU as the
checkpointed process. The current scope is single-GPU Qwen3-0.6B.

## Ownership

The checkpointed process owns tensor objects, StorageImpls, aliases, views,
model topology, virtual addresses, and one canonical record per raw allocator
mapping. The sidecar owns each allocation ID's aligned physical allocation and
retained export handle. The sidecar also exposes a process-incarnation nonce
and physical GPU UUID.

`hello`, `allocate`, `export`, and `free` are the complete RPC protocol.
Allocation IDs are generated by the client. Repeating `allocate` with the same
ID and size or repeating `free` is safe after response loss.

## Model loading and normalization

The dedicated vLLM worker patches the existing
`BaseModelLoader.load_model`. Model construction, normal weight loading, and
model post-processing run in one temporary GMS-backed Torch MemPool. The
normal vLLM load format and loader remain unchanged.

Immediately after `BaseModelLoader` returns, and outside the GMS pool, V1
groups live module tensors by StorageImpl. Every complete GMS-backed storage
that contains no Parameter is cloned once through the default CUDA allocator.
Existing tensor objects are swapped onto the cloned storage without changing
their identities, aliases, views, storage offsets, shapes, or strides. A
non-Parameter tensor that shares a StorageImpl with a Parameter remains on the
Parameter storage.

V1 then synchronizes, drops the exact temporary MemPool, and runs Python
garbage collection. Torch's allocator free callback releases wholly inactive
raw GMS mappings. V1 does not retain displaced source owners, prune mappings,
or persist tensor metadata. A raw mapping can retain inactive capacity when a
live Parameter block keeps the segment active.

After pool destruction, V1 validates storage-level ownership:

- every CUDA Parameter storage is fully contained in one surviving GMS mapping;
- every surviving GMS mapping contains a CUDA Parameter storage;
- a named CUDA buffer or other discovered module tensor cannot overlap a
  surviving GMS mapping unless it has the exact Parameter StorageImpl.

This normalization completes before memory profiling, KV-cache allocation,
warmup, or CUDA graph capture. All later vLLM runtime tensors, sampler state,
KV cache, and other non-Parameter allocations use the normal allocator and are
handled by Dynamo CUDA Snapshot.

The process-global vLLM `WorkspaceManager` is the one explicit exception to
ambient default allocation. Growth is routed into a retained native/default
Torch MemPool so an outer allocation context cannot route workspace backing
into GMS. No second GMS pool is created.

## Snapshot lifecycle

Surviving GMS mappings are Parameter backing. They are read-write during model
load. On sleep, V1:

1. synchronizes the GPU;
2. changes every surviving mapping to read-only;
3. unmaps and releases every local import exactly once while preserving virtual
   addresses, allocation IDs, and sidecar backing;
4. disconnects the allocation client only after sleep succeeds.

Before wake exports or imports anything, the client opens a fresh Unix-domain
socket and verifies the stored sidecar nonce and GPU identity. Wake then
exports the same allocation IDs and maps them read-only at the same virtual
addresses. It never creates fresh backing.

Cleanup failures are fail-stop. The manager retains ownership evidence when a
resource cannot be proved released and continues independent cleanup where
possible.

## vLLM

Select the worker explicitly and keep the normal load format:

```text
python -m dynamo.vllm ... \
  --worker-cls gpu_memory_service.v1.integrations.vllm.worker.GMSV1Worker
```

The worker selects the V1 sleep backend and leaves vLLM's native `weights` and
`kv_cache` allocator scopes on the normal/default allocator. Only whole-engine
level-1 sleep and untagged wake are supported. A failed lifecycle transition
terminates the worker process.

This milestone has no custom model loader, load format, model manifests, meta
model, tensor metadata, restore materialization, compatibility framework,
distributed hooks, or vLLM PR #46877 hooks.

## Commands and tests

Start the sidecar with:

```text
gms-v1-server --device 0 [--socket-path PATH]
```

Focused lifecycle, storage normalization, RPC, and vLLM integration tests live
in `lib/gpu_memory_service/tests/test_v1_*.py`.
