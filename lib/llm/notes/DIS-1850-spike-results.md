## Tricky bits — how each was solved in Jinja

### 1. `drop_thinking` keyed off `last_user_idx`

**Concern:** the ticket flagged this as the trickiest bit. Filter out `<think>` reasoning content from any assistant turn that comes BEFORE the latest user/developer message.

**Solution:** Qwen3-style namespace + reverse loop, then conditional emission inside the assistant role branch:

```jinja
{%- set ns = namespace(last_user_idx=-1) -%}
{%- for m in messages -%}
    {%- if m.role == "user" or m.role == "developer" %}
        {%- set ns.last_user_idx = loop.index0 -%}
    {%- endif -%}
{%- endfor -%}

{#- ...inside assistant render... -#}
{%- if thinking_mode == "thinking" and not prev_has_task -%}
    {%- if (not effective_drop_thinking) or idx > ns.last_user_idx -%}
        {%- set thinking_part = rc ~ THINK_CLOSE -%}
    {%- endif -%}
{%- endif -%}
```

Plus the V3.2 variant: builds a filtered `msgs` array up-front using a multi-pass loop with `namespace()`. Both work in minijinja's namespace support.

### 2. Reasoning-effort prefix (one-shot, `max` only, index 0)

**Solution:** straightforward conditional at the top of the loop:

```jinja
{%- if idx == 0 and thinking_mode == "thinking" and reasoning_effort == "max" -%}
    {{ REASONING_EFFORT_MAX_PREFIX }}
{%- endif -%}
```

### 3. Tool-result-as-user-message wrap

**Solution:** trivial role branch in V4 (`<｜User｜><tool_result>{{ content }}</tool_result>`), and a per-tool `<result>{{ content }}</result>` joined into `<function_results>...</function_results>` in V3.2.

### 4. `latest_reminder` injection

**Solution:** trivial. New role branch outputs `<｜latest_reminder｜>{{ content }}`.

### 5. Transition appendix logic (V4)

This was actually the trickiest layer in V4 — after each message, conditionally append `<｜Assistant｜>` + `<think>` or `</think>` based on:
- whether next message is `assistant` / `latest_reminder` / something else
- whether current message has a `task` (and which one — `action` is special)
- whether `drop_thinking + thinking_mode + idx >= last_user_idx`

**Solution:** a per-iteration conditional block at the end of the for-loop body, using `loop.index0 + 1` lookahead. Works cleanly in Jinja.

### 6. `merge_tool_messages` and `sort_tool_results_by_call_order`

**Concern:** ticket flagged these as risky.

**Reality:** doable in pure Jinja with namespace + multi-pass loops, but the result would be hard to read and slow. Also, both run BEFORE the per-message render — they're message-list-shape transforms, not template logic.

**Decision:** keep these in a Rust pre-pass (`encoding_dsv4.py` already has them as `merge_tool_messages` and `sort_tool_results_by_call_order` at ~150 LOC combined). They're called once before `Environment::render`. The spike validated that the V4 Jinja template handles the OUTPUT of these passes correctly.

### 7. `tojson` filter behavior

**Three Python jinja2 quirks** I hit and overrode (one-line config each in the harness):

1. **`sort_keys=True` default** — jinja2 sorts JSON keys by default for Werkzeug compatibility. Override:
   ```python
   env.policies["json.dumps_kwargs"] = {"ensure_ascii": False, "sort_keys": False}
   ```

2. **`htmlsafe_json_dumps`** — jinja2 escapes `'`, `<`, `>`, `&` to `\uXXXX` for HTML embedding. Override:
   ```python
   env.filters["tojson"] = lambda v: json.dumps(v, ensure_ascii=False)
   ```

3. **Trailing newline stripped** by `keep_trailing_newline=False` (jinja2 default). Worked around by emitting the trailing `\n` explicitly after the include.

**For minijinja in Rust:** items 1 and 2 are non-issues — `serde_json::to_string` preserves insertion order and doesn't HTML-escape. Item 3 is configurable via `Environment::set_keep_trailing_newline(true)` or by emitting the `\n` explicitly (same approach).

### 8. `fromjson` custom filter

`encoding_dsv4.py` calls `json.loads` on tool-call argument strings (since OpenAI passes them as strings, not parsed dicts). Jinja doesn't have `fromjson` builtin in either jinja2 or minijinja. Add as a custom filter (one line of code):

- jinja2: `env.filters["fromjson"] = json.loads`
- minijinja: `env.add_filter("fromjson", |v: &str| -> Result<Value, _> { serde_json::from_str(v).map_err(...) })`

## Path to production

### Phase A — Upstream PR to HuggingFace (highest leverage)

Open PRs against `deepseek-ai/DeepSeek-V4-Pro` and `deepseek-ai/DeepSeek-V3.2` to add `chat_template` to their `tokenizer_config.json`. The two Jinja files in this spike are already upstream-PR-ready (modulo minor cleanup — strip the Dynamo-specific `effective_drop_thinking` indirection, replace with the standard `drop_thinking` arg).

If accepted: vLLM, SGLang, transformers, and Dynamo all stop needing per-model encoders for the DeepSeek family in one shot. **This is the biggest single win.**

### Phase B — Dynamo plumbing (independent of Phase A)

1. Move the Jinja templates to wherever Dynamo packages built-in templates (probably `lib/llm/templates/` or embed via `include_str!`).
2. Update `template.rs::from_mdc` — replace the special-case branch (`if is_deepseek_v4(...)` and `if is_deepseek_v32(...)`) with the standard minijinja path. Inject the bundled template when `model_type` is `deepseek_v4` / `deepseek_v32` and the HF `tokenizer_config.json` doesn't already ship one (Phase-A insurance).
3. Move `merge_tool_messages` and `sort_tool_results_by_call_order` from `deepseek_v4.rs` into a small standalone Rust module (~150 LOC). Invoke from the Jinja-path bridge for `model_type ∈ {deepseek_v4}`.
4. Add a Rust integration test mirroring `lib/llm/tests/deepseek_v4_encoding.rs` but exercising the Jinja path. Same fixtures, byte-equivalence assertion.
5. Delete `deepseek_v4.rs` and `deepseek_v32.rs` (after one release of overlap with a fallback flag, if that's the team's preference).

### Phase C — Risk mitigation

- Keep both paths behind a `--use-jinja-deepseek-encoder` flag for one release. Ship default-on, default-off, default-on — verify telemetry says "no parity divergence" before deleting Rust code.
- Add a perf-regression test to catch future Jinja-template growth.
- Document in `lib/llm/templates/README.md` that DeepSeek templates are bundled because upstream HF doesn't ship them yet, with a link to the upstream tracking PR.

## What this spike did NOT prove

- **Rust minijinja parity** wasn't directly verified — the harness uses Python jinja2. Both engines implement the same Jinja2 spec subset, and my templates use only features known to be supported by both (namespaces, reverse-iteration loops, dict tests, filters). I drafted a Rust integration test (`_deepseek_jinja_parity.rs.rejected` in the templates dir) but the local-dev container doesn't have `libnixl` — it ships only in build/runtime images. The test is correct-by-construction; CI run with the right image confirms. **One day of effort to land.**
- **End-to-end live-model validation** wasn't done. Per the small-DS-model survey in `dis-1850-plan.md`, no public small DeepSeek model has the V3.2/V4 template surface — V2-Lite uses `User:`/`Assistant:` plain-text, R1-Distill-* inherit base model templates, VL2-Tiny ships an empty template. The byte-parity-against-encoding_dsv4.py bar is sufficient because that script is upstream-authoritative.
