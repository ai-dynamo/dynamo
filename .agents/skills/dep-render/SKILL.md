## Purpose

Publish a DEP as a page on the Dynamo docs site (Fern, under the **Proposals**
tab) and mirror its GitHub review read-only, one-way. The DEP markdown itself is
owned in **`ai-dynamo/enhancements`**; this skill is the docs-site render +
mirror layer only. The rendered page pulls three GitHub surfaces at load time:
inline PR line comments (anchored on the quoted text), the PR conversation
("Revision discussion"), and the tracking-issue thread ("Design discussion"),
each deep-linking back to GitHub to reply. Bots/CI (`[bot]`, `copy-pr-bot`,
`github-actions`) are filtered out. The page never posts; GitHub stays the
source of truth.

Ground everything in `docs/proposals/README.mdx` (the model + the "Commenting
on a DEP" section) and `docs/proposals/0001-dep-process.mdx` (Implementation
Details + Deferred to Implementation). Do not restate the whole model here.

The pieces this layer is built from:

- `fern/components/DepMetadata.tsx` — status pill + metadata card.
- `fern/components/PrInlineComments.tsx` — the SSR mount div.
- `fern/js/dep-pr-comments.js` — runtime that reads `owner`/`repo`/`pr`/`issue`/`path`
  data-attrs and fetches the three GitHub REST sources.
- `fern/scripts/sync_deps.py` + `fern/scripts/deps.json` — build-time content
  sync, emitting generated mdx under `docs/proposals/_generated/` (gitignored).
- `fern/js/dep-status-pills.js` fed by build-time `fern/js/dep-status-data.js`
  (generated by `sync_deps.py`) — the right-aligned sidebar status pill; its
  colors mirror `DepMetadata.statusVariant()`.

Cross-repo default is `owner="ai-dynamo" repo="enhancements"`.

## When to Use

When a DEP needs a page on the docs site, or an existing DEP page needs to be
refreshed (new PR/issue wired, status banner updated, synced from upstream).

## Two Ways to Render

### Option A — Build-Time Sync (preferred; no DEP body copied here)

Add one entry to `fern/scripts/deps.json`. The body is fetched from
`ai-dynamo/enhancements` on every Fern build and emitted to
`docs/proposals/_generated/<output>.mdx` (gitignored — the `ai-dynamo/dynamo`
repo never commits a DEP body).

```json
{
  "output": "dep-<slug>",
  "dep": "NNNN",
  "subtitle": "One-line description rendered under the H1.",
  "pr": 123,
  "tracking_issue": 120,
  "tracking_issue_url": "https://github.com/ai-dynamo/enhancements/issues/120",
  "source": {
    "owner": "ai-dynamo",
    "repo": "enhancements",
    "ref": "main",
    "path": "deps/NNNN-slug.md"
  }
}
```

Then add the nav entry to the `proposals` tab in `docs/index.yml`:

```yaml
      - page: "DEP-NNNN: <Title>"
        path: proposals/_generated/dep-<slug>.mdx
        slug: dep-<slug>
```

`sync_deps.py` wraps the synced body with `<DepMetadata>` (fields parsed from
the upstream `**Key**: Value` metadata block, status defaulting to Draft) and a
`<PrInlineComments>` mount pointed at the source repo/PR, demotes headings one
level, and escapes stray `<digit` sequences.

### Option B — Hand-Author the MDX Page

Copy `docs/proposals/TEMPLATE.mdx` to `docs/proposals/NNNN-slug.mdx`, fill it
in, and mount the two components:

```mdx
import { DepMetadata } from "@/components/DepMetadata";
import { PrInlineComments } from "@/components/PrInlineComments";

<DepMetadata
  dep="NNNN"
  status="Draft"
  category="Architecture"
  owningSig="SIG-<Area>"
  authors="[@handle](https://github.com/handle)"
  owner="ai-dynamo"
  repo="enhancements"
  pr={123}
/>

> [!WARNING]
> **Status: Draft.** This proposal is under active discussion and is not an accepted decision.

{/* ...DEP body... */}

<PrInlineComments
  owner="ai-dynamo"
  repo="enhancements"
  pr={123}
  issue={120}
  path="deps/NNNN-slug.md"
/>
```

Set `path` to the DEP's repo-relative path so a multi-file PR only mirrors
comments on this DEP. Omit `issue` to show the "open a tracking issue" empty
state. Add the matching nav entry to `docs/index.yml` (path
`proposals/NNNN-slug.mdx`, stable `slug: NNNN-slug`).

## Set the Status Banner

Open the page with a banner matching the DEP's `status`, per the convention in
`docs/proposals/README.mdx`:

| Status | Banner admonition |
|--------|-------------------|
| Draft | `> [!WARNING]` |
| Under Review | `> [!IMPORTANT]` |
| Accepted / Implemented | `> [!NOTE]` |
| Rejected / Deferred / Replaced | `> [!CAUTION]` |

The right-aligned **sidebar status pill is generated automatically** — you do
not hand-write it. `sync_deps.py` reads the status (from the synced upstream
`**Status**:` field, or from the hand-authored `<DepMetadata status="...">`)
and writes `fern/js/dep-status-data.js`, which `fern/js/dep-status-pills.js`
consumes on boot. Keep the banner and the `status` in sync; advancing the
status itself is the `dep-update` skill (upstream), which propagates here on the
next build.

## Build and Check

```bash
python3 fern/scripts/sync_deps.py       # regenerate _generated/*.mdx + dep-status-data.js
fern check                              # validates the docs graph (CI runs this too)
fern docs broken-links                  # link check
```

`sync_deps.py` runs first in CI (the Fern preview and the pre-merge
`fern check` jobs) so `docs/index.yml` entries pointing at `_generated/*.mdx`
resolve. Use `--only <slug>` to sync a single entry and `--dry-run` to preview.

## Verify the Preview

On the Fern PR preview (commented on the PR as a `.../dev` URL), confirm:

- The page resolves at the stable **`/proposals/<slug>`** URL.
- The **sidebar status pill** shows and matches the on-page `<DepMetadata>`
  pill color.
- The **three-surface comment mirror** renders: inline review anchored on the
  quoted text, "Revision discussion" (PR conversation), and "Design discussion"
  (tracking issue), with bots/CI filtered out.
- It renders correctly in **light and dark mode**.

## Notes and Deferred Work

- The comment fetch is **unauthenticated**, so it is subject to GitHub's
  ~60-requests/hour/IP limit; on a 403 the affected section degrades to a notice
  plus a link to the thread.
- **Build-time comment baking** (to remove that limit on high-traffic
  production pages) is deferred work called out in DEP-0001; today the mirror
  fetches live in the browser.
- The `deps.json` manifest currently pins a source `ref` (branch) rather than a
  per-DEP SHA — a known caveat tracked alongside DEP-0001 acceptance.
- The generated dir `docs/proposals/_generated/` is gitignored; never commit a
  synced DEP body. Only the manifest, `sync_deps.py`, and nav entries live here.
